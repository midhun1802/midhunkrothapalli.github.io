01
System Architecture
How LabWare is structured — the application tiers, database layer, deployment models, and what each component does. The foundation everything else sits on.
▸ Click any node to expand details
🌐 Web Browser Client

LabWare LIMS v7+ delivers a full HTML5 web interface accessible from any modern browser — no desktop client install required. This is the primary access method for most users in current deployments.

  • Zero-install: Users access LIMS via HTTPS URL. Browser renders the full application. Only supported browsers (Chrome, Edge, Firefox — versions defined in IQ) are validated.
  • Responsive layout: Adapts to screen size. Larger screens show full sample/result grids; tablets show simplified data entry views.
  • Session management: Inactivity timeout configurable (typically 15–30 min for GxP compliance). Session expiry forces re-authentication — Part 11 requirement.
  • Audit trail: All actions taken through the web client are captured in the system audit trail with user ID, timestamp, and IP address.
OQ test: Verify that a session automatically expires after the configured inactivity period and requires re-login before any further GxP actions can be performed.
🖥️ Thin Client (LabWare Desktop)

The legacy LabWare thick/thin client is a Windows application used in older deployments or for specific functions not yet migrated to the web UI (e.g., advanced admin configuration, specific instrument interface utilities). Being phased out in favour of the web interface in v7+.

  • Connects to the application server via TCP/IP on a defined port
  • Must be version-matched to the server — mismatches cause connection failures and are a common post-upgrade issue
  • IQ must document the installed version on every workstation that uses the thin client
📱 Mobile / Tablet Access

LabWare supports mobile access for field sampling and barcode-driven workflows. Key use cases:

  • Field sample login: Lab tech scans a barcode at the sampling point. Mobile app creates the sample record with GPS-tagged location and timestamp — contemporaneous recording (ALCOA "C").
  • Sample receipt confirmation: Receiving lab scans incoming samples to confirm receipt. Status updates in real-time.
  • Stability pull confirmation: Analyst scans stability samples at pull time — system records the actual pull date vs scheduled pull date for deviation tracking.
Validation note: Mobile devices used for GxP data entry must be included in the IQ (device model, OS version, browser version). The mobile workflow is a separate OQ test scope from the desktop workflow.
🔌 REST API Access

LabWare v7+ exposes a RESTful API for programmatic access — used for system integrations (ERP, ELN, instruments) and automation.

  • Authentication: OAuth 2.0 or API key. Service accounts used for integrations must be named (not shared), have minimal permissions, and be included in access reviews.
  • Key endpoints: Sample creation, result submission, status queries, report retrieval, user management.
  • Audit trail: API-sourced changes carry the service account user ID in the audit trail. Distinguishable from human-entered records — ALCOA "Attributable" requirement.
Validation scope: API integrations are validated interfaces. Each endpoint used in GxP workflows needs an IQ (connectivity established) and OQ (data sent via API appears correctly in LIMS with proper audit trail attribution).
⚙️ LabWare Application Server

The application server is the core processing layer — it runs on WebLogic or Apache Tomcat (depending on version and deployment). All business logic, workflow routing, and LBMS script execution happens here.

  • Clustered deployment: Production systems use multiple app server nodes in a cluster for high availability. Load balancer distributes user sessions across nodes.
  • GxP implication: If one node fails, user sessions may be interrupted. The recovery behaviour (session persistence, re-authentication) must be tested in OQ under a failover scenario.
  • JVM tuning: Heap size and thread pool settings affect performance under high concurrent user load. Performance testing (concurrent users, response time under load) is part of PQ for high-throughput labs.
  • Environments: A separate DEV, QA/TEST, and PROD app server instance is required for a validated LabWare deployment. Changes promoted DEV → QA (test) → PROD only after validation sign-off.
📝 LBMS Engine (LabWare Basic for Microsoft)

LBMS is LabWare's built-in VBA-like scripting engine. It runs inside the application server and executes custom logic in response to system events.

  • Event triggers: On-Save, On-Status-Change, On-Login, On-Approval, On-Report-Generation
  • Use cases: Complex multi-component calculations, dynamic test assignment logic, custom OOS rules, notification routing, instrument data parsing, report population
  • GAMP Category: Any LBMS script written by your team is GAMP Cat 5 (custom code) — requires code review, unit testing, and formal OQ validation. Not covered by LabWare's vendor qualification.
Critical admin risk: LBMS scripts accumulate over years and are often undocumented. Every script in a validated vault must be inventoried, version-controlled (source code management), and tested. A script inventory audit is mandatory at every periodic review. Undocumented scripts running in production are a critical finding.
📊 Report Server — Crystal Reports & CoA Generation

LabWare uses SAP Crystal Reports as its primary reporting engine. The report server renders reports on demand from LIMS data.

  • Certificate of Analysis (CoA): The most GxP-critical report. Generated at batch release — contains all test results, specifications, and pass/fail status for a product batch. Must be validated to confirm it accurately reflects the LIMS data and cannot be manually edited after generation.
  • Audit trail reports: Regulators request LIMS audit trail reports during inspections. Pre-built and validated audit trail report templates are essential.
  • Label printing: Sample labels, stability rack labels, and reagent labels generated via Crystal Reports and sent to barcode printers on the network.
  • Report validation: Each GxP report is a validated function. OQ tests: correct data populates report, calculated fields are accurate, report cannot be modified after generation, and the report format (field positions, headers, footers) matches the approved template.
🗄️ Oracle / SQL Server Database

LabWare's primary data store. Oracle is most common in large enterprise deployments; SQL Server used in some smaller/mid-size implementations.

  • Schema: LabWare installs a complex schema with hundreds of tables. Core tables: SAMPLE, TEST, COMPONENT, RESULT, AUDIT_TRAIL, USER_MASTER, PRODUCT, SPECIFICATION. Never modify the schema directly — all customisation goes through LabWare's admin interface or approved DB procedures.
  • Direct DB access control: Only DBAs should have direct SQL access to production. All admin changes through the LIMS UI. Direct DB queries for reporting should use read-only accounts. Any direct DB modification in production is an emergency-only action requiring retrospective change control and is visible to auditors.
  • IQ requirements: Oracle version, character set, tablespace sizing, backup schedule, and DBA access list all documented in IQ.
Audit exposure: Oracle's audit trail (v$aud table) captures direct DB-level access. If an inspector requests this and finds direct SQL updates to GxP tables outside of the LIMS workflow, it is a critical data integrity finding. All data changes must flow through the application layer.
📁 File Storage — Attachments & Raw Data

LIMS stores binary files (instrument raw data files, PDFs, images, spectra) in a network file store linked to result and sample records.

  • Each attachment is linked to a specific result, sample, or experiment record
  • File integrity verification: LabWare stores a hash (checksum) of each attached file. Verification can confirm the file has not been altered since attachment.
  • Backup must cover the file store AND the database — restoring the DB without the file store results in broken record links
  • Retention policy: files must be retained for the same period as their linked GxP records (batch life + applicable regulatory retention)
🔒 Archive & Disaster Recovery

GxP systems require documented, tested backup and DR procedures as part of validation.

  • Backup frequency: Daily full backup minimum; transaction log backups every 1–4 hours for near-zero RPO
  • Backup verification: Monthly restore test from backup — not just a backup job completing successfully, but verifying the restore actually produces a working system
  • DR instance: High-criticality LIMS systems (batch release) should have a DR instance that can be activated within the defined RTO (Recovery Time Objective). RTO must be defined in the URS and tested in IQ/OQ.
  • Audit trail of backup activity: Backup job logs are compliance evidence. Include in periodic review.

LabWare Data Model
The core object hierarchy — how Sample, Test, Component, and Result relate, and where specifications and products fit in. This is the mental model you must have cold.
▸ Click any object to expand what it contains and how it relates
🏷️ Product / Matrix — The Configuration Foundation

The Product (also called Matrix in LabWare) is the master configuration record that defines everything about a specific drug product or material. Every sample that ever enters the LIMS traces back to a Product record.

What a Product record contains:

  • Identity: Product name, product code, NDC (National Drug Code), CAS number, active ingredients, dosage form, strength
  • Matrix type: Raw Material, API, Drug Product, In-Process, Finished Product, Stability, Reference Standard — determines which workflows and validations apply
  • Test Plan: The list of analyses required for this product (Identification, Assay, Dissolution, Microbial, Related Substances, etc.) and the conditions under which each is required (Release testing vs. Stability T=0 vs. Stability T=6mo)
  • Specifications: For each test component — lower limit, upper limit, target, units, decimal places, rounding rules, OOS threshold
  • Storage conditions: Temperature, humidity, light requirements — relevant for stability scheduling
Config governance: Product records are GxP configuration. Creating or modifying a Product record in production must go through change control. New product additions need documented approval. Specification changes must be traced to an approved specification document and tested in OQ before production.
🧪 Sample — The Primary GxP Record

The Sample is the top-level GxP record. Every analytical result in LabWare belongs to a Sample. The Sample is what QA ultimately approves for batch release.

Key sample fields:

FieldDescriptionGxP Significance
Sample IDSystem-generated unique identifier. Must be unique across all samples — never reused.ALCOA Attributable — uniquely identifies which physical sample this data belongs to
Login DateDate/time the sample was registered. System-generated, cannot be backdated.ALCOA Contemporaneous — record created at time of receipt
Batch / Lot NumberManufacturing batch this sample representsLinks to batch release decision
Sampling DateWhen and where the physical sample was takenTraceability to the manufacturing event
StatusU / I / C / A / D / X (lifecycle states)Controls access and edit permissions at each stage
Expiry DateSample retention expiryDrives disposal workflow and retention compliance
🔬 Test — The Analytical Procedure Unit

A Test represents one analytical method assigned to a sample. A single sample may have many tests (e.g., Assay, Dissolution, Identification, Microbial Limits, Related Substances — all separate tests on the same sample).

  • Test template: Each test is based on a template that defines its components, calculation logic, worksheets, and equipment required
  • Test status: Mirrors sample status — Scheduled, In Progress, Completed, Approved. Tests can be individually approved before the parent sample is approved (useful when different analysts handle different tests)
  • Analyst assignment: Tests are assigned to specific analysts (or analyst groups). Only the assigned analyst can enter results unless an override is used (which creates an audit trail entry)
  • Due date: Tests have due dates calculated from the sample login date based on configured turnaround time. Overdue tests trigger notification workflows
⚗️ Component — The Individual Measurable Parameter

A Component is a single measurable parameter within a test. It is the level at which specifications are defined and at which OOS is evaluated.

Example — HPLC Assay test:

HPLC Assay Test ├── Component: Active Ingredient % Spec: 97.0 – 103.0 % ├── Component: Related Substance A % Spec: ≤ 0.1 % ├── Component: Related Substance B % Spec: ≤ 0.2 % └── Component: Total Impurities % Spec: ≤ 0.5 %

Each component has: data type (numeric, text, pass/fail), units, specification type (between, less than, greater than, equals), decimal places for display and rounding, and whether it contributes to the OOS flag on the parent test.

📋 Result — The Atomic GxP Data Unit

The Result record is the most important object in LabWare from a regulatory perspective. It holds the actual data that supports every quality and batch release decision.

  • Entered value: The raw value as entered by the analyst
  • Calculated value: If a calculation formula is applied (LBMS script), the calculated result is stored separately from the entered value — both retained
  • OOS flag: Boolean — set automatically by the system when the entered/calculated value is outside specification. Cannot be manually cleared without an OOS investigation workflow
  • Entered by / date: System-recorded — who entered this value and when. Cannot be modified.
  • Reviewed by / date: Supervisor review — captured as a separate event
  • Approved by / date: QA approval with e-signature
  • Data correction history: If a result is corrected post-entry, the original value, new value, reason, corrector, and QA approver are all preserved in the result's correction history
ALCOA+ compliance check: Attributable (who entered) ✓ · Legible (stored digitally) ✓ · Contemporaneous (system timestamp) ✓ · Original (first entry preserved) ✓ · Accurate (OOS check against spec) ✓
📅 Stability Records

Stability samples are a special Sample subtype with a linked Stability Protocol that defines timepoints (T=0, T=1mo, T=3mo, T=6mo, T=12mo, T=24mo), storage conditions (25°C/60%RH, 40°C/75%RH), and test requirements at each timepoint. LabWare auto-schedules pull dates and generates alerts when pulls are due. Stability results flow through the same Sample→Test→Component→Result hierarchy but are tracked against the protocol schedule. OOT (Out of Trend) detection uses trend analysis across timepoints. See Stage 11 for full detail.

🚨 OOS Record — Auto-Generated Linkage

When a Result's OOS flag is triggered, LabWare can be configured to automatically create a linked OOS Investigation record in the QMS (or internal OOS module). The OOS record carries the parent Sample ID, Test, Component, specification limits, and the actual failing value. The OOS investigation must be resolved (or the result invalidated with documented justification) before the parent sample can move to Approved status. See Stage 09 for the complete OOS workflow.

Security, Roles & Access Control
How users are authenticated, how roles restrict what they can see and do, and how 21 CFR Part 11 separation-of-duties is enforced in configuration.
▸ Click any role tier to see permissions and Part 11 implications
🔑 System Admin — Most Powerful, Most Controlled

What System Admin can do: Everything. Create/modify products, tests, specifications. Create/disable user accounts. Modify access profiles. Execute LBMS scripts. Run direct DB queries (if DB access granted). Access all menus including configuration menus hidden from regular users.

Critical governance rule: System Admin access must be granted only to named individuals — typically 2–4 people maximum. No shared admin accounts. Admin accounts must be reviewed quarterly. Separation of duties: the person who makes a configuration change should not be the same person who approves the change control authorising it. Every admin action is captured in the admin audit trail — inspectors will ask to see it.
  • Admin account access requires business justification and QA approval
  • Admin actions in production must correspond to approved change controls
  • Former employee admin accounts must be disabled within 24 hours of departure
✅ QA Approver Role

Key permissions:

  • Can move samples from Completed → Approved status (requires e-signature)
  • Can approve OOS investigation records
  • Can approve data correction requests
  • Can generate and approve Certificates of Analysis
  • Can view all samples across all products and sites
  • Cannot: Enter or modify results (separation from analytical work)
Part 11 enforcement: The access profile for QA Approver blocks UPDATE permission on RESULT table records where the ENTERED_BY field matches the current user. System-level enforcement of self-approval prohibition.
👁️ QC Supervisor Role

Key permissions:

  • Can review results entered by analysts (review ≠ approval — separate action)
  • Can perform Phase 1 laboratory investigation for OOS events
  • Can authorise analyst overrides (e.g., use a different instrument than assigned)
  • Can move samples from In Progress → Completed (supervisor sign-off that all tests are entered)
  • Can view all samples in their assigned department/site
  • Cannot: Approve samples for batch release (QA role only), modify specifications
⚗️ QC Analyst Role

Key permissions:

  • Can enter results on tests assigned to them
  • Can view samples assigned to their login group
  • Can attach raw data files (instrument outputs) to result records
  • Can print worksheets and labels
  • Cannot: Approve their own results, modify specification limits, create/delete samples, access admin menus

Status-based restriction: Once a result reaches Completed or Approved status, the Analyst role loses UPDATE permission on it. Any post-completion correction requires a formal data correction workflow with Supervisor + QA authorisation.

📥 Login Clerk Role

Narrowest operational role. Can only create new sample records (log samples into the system) and print labels. Cannot enter results, cannot view test data, cannot approve anything. Used in labs where sample receipt is handled by a separate team from analytical testing. Ensures that the person logging the sample cannot also manipulate the analytical data — separation of duties at the sample receipt stage.

👁️ Read-Only Role

View-only access to approved records. Typically used for: regulatory affairs staff reviewing test data for submissions, external auditors granted temporary access, manufacturing personnel checking batch release status. Cannot create, edit, or delete any record. All access is still audit-trailed — even views of sensitive records are captured, which can be reviewed to detect unauthorised access patterns.

Sample Login
The moment a physical sample enters the LIMS. This is where the chain of custody begins and the GxP record is created. Every downstream action traces back to this event.
▸ Click each step in the login flow
🔍 Step 1 — Scan or Manual Entry

Sample login can be initiated in three ways:

  • Barcode scan: Pre-printed sample container barcode scanned at receipt — automatically populates the Sample ID field and may pre-fill product data if the barcode carries a GTIN or product code. Fastest and most accurate method — reduces transcription errors.
  • Manual entry: Analyst types the sample information. More error-prone — validation rules (duplicate checks, format rules) compensate.
  • Batch login: For high-throughput labs, multiple samples logged simultaneously via a CSV upload or a batch login screen. Each record still gets a unique Sample ID and independent audit trail.
Best practice: Barcode-driven login is preferred for all GxP labs. It enforces ALCOA "Attributable" — the barcode carries the sample identity from the point of origin and eliminates name/ID transcription errors that can create data integrity questions.
📝 Step 2 — Login Form Completion

The login form is configured in LabWare admin. Fields and their mandatory status are defined per product type or matrix. Common fields:

Always Required
Product / Matrix • Batch/Lot Number • Sample Type (Release/Stability/In-Process) • Collection Date • Quantity • Unit of Measure • Received By
Conditionally Required
Sampling Location (required for environmental) • Stability Protocol (required for stability type) • Expiry Date (required for reference standards) • Storage Condition (for temp-sensitive)

LBMS scripts can enforce additional business rules at login — for example, blocking login of a new batch if a previous batch of the same product has an open OOS investigation.

✔️ Step 3 — System Validation Rules

Before the sample record is created, LabWare runs a series of validation checks:

  • Mandatory field check: All required fields populated. Configurable per product/matrix.
  • Duplicate check: Is there already an active sample with this Lot Number and Product? Configurable: warn or block. Prevents the same batch being tested twice accidentally.
  • Format validation: Lot number format matches the expected pattern (e.g., must be 8 characters alphanumeric). Date fields within valid range. Numeric fields within expected range.
  • Reference data lookup: Product code exists in the system. If entered product is not configured — block login and alert admin.
Uniqueness of identifiers (WeightID / SampleID): Any field intended to be a unique identifier MUST have a system-enforced uniqueness constraint — both at the application validation layer and at the database unique index level. The 2024 LabX WeightID duplicate incident was caused by exactly this gap. Configure and validate uniqueness on ALL identifier fields during initial configuration.
⚡ Step 4 — Automatic Test Assignment

The moment the sample is saved, LabWare reads the product's Test Plan and automatically creates Test records for every analysis required for this sample type and purpose.

Example — Drug Product Tablet, Release Testing:

Product: Drug Product Tablet 500mg · Purpose: Release Auto-created tests: TEST-001 Identification (IR) Assigned to: QC Analyst Group TEST-002 Assay (HPLC) Assigned to: HPLC Analyst Group TEST-003 Related Substances (HPLC) Assigned to: HPLC Analyst Group TEST-004 Dissolution Assigned to: Dissolution Team TEST-005 Microbial Limits Assigned to: Microbiology Lab TEST-006 Water Content (KF) Assigned to: QC Analyst Group TEST-007 Appearance Assigned to: QC Analyst Group

Each test gets a due date, an analyst assignment, and a Scheduled status. The QC scheduling board now shows this sample as a live work item.

🏷️ Step 5 — Label Printing

On successful login, LabWare triggers label printing for the physical sample container(s). The label contains:

  • Sample ID (human-readable + 1D/2D barcode)
  • Product name and lot number
  • Login date
  • Storage condition indicator (colour-coded or text)
  • Lab identifier

Label format is defined in a Crystal Reports template validated as part of the OQ. The label template cannot be modified without a change control. Labels are the physical link between the database record and the physical sample — if a label is lost or damaged, a reprint workflow with audit trail entry is required.

Test Assignment & Scheduling
How tests get to the right analysts, how priority and due dates are managed, and how the QC scheduling board works.
🗓️ Test Assignment Architecture

LabWare test assignment operates on a Group → Analyst model:

  • Login Group assignment: Tests are initially assigned to a Login Group (e.g., "HPLC Team", "Microbiology"). Any analyst in that group can pick up the test.
  • Individual pickup: Analyst selects a test from their group's worklist and "claims" it — the test is now assigned to them specifically. Their initials appear on the test record. Other analysts can no longer edit results on that test without an authorised override.
  • Due date calculation: System calculates due date = Login Date + configured TAT (Turnaround Time) for the test type. TAT is defined per test in the Product test plan. Overdue tests are highlighted on the scheduling board and trigger escalation notifications.
  • Priority override: Supervisors can mark a sample as Priority/STAT. Priority samples float to the top of every analyst's worklist and receive a visual flag.
Scheduling Board View
Real-time grid showing all open tests, grouped by analyst or test type, with colour-coded status (On Time / Due Soon / Overdue). Supervisors use this to balance workload and spot bottlenecks.
Capacity Planning
LabWare reports show incoming sample volume vs. available analyst capacity. Used for resource planning — scheduling overtime or requesting temp support before TAT is at risk.

Worksheets & Instrument Scheduling
How batches of tests are grouped for instrument runs, how worksheets organise multiple samples on one analytical run, and how QC standards and blanks are managed.
📋 Worksheet Architecture

A Worksheet in LabWare groups multiple test records into a single analytical run — matching how a real lab instruments work (you don't run one sample per HPLC injection, you run a sequence of 20–30).

  • Worksheet template: Pre-configured in admin. Defines: how many samples per run, what QC positions are required (System Suitability, Calibration Standard, Blank, Control), the injection sequence order, and what fields must be filled before the run can be approved.
  • Auto-population: When an analyst creates a worksheet, LabWare populates it with pending tests of the matching type from the queue — up to the capacity defined in the template. Analyst reviews the population and can add, remove, or reorder.
  • QC positions (System Suitability / Calibration): Mandatory positions for reference standards and blanks are auto-inserted per the template. Results for these positions are evaluated against QC acceptance criteria before any sample results can be finalised. If QC fails → all sample results in the run are held pending investigation.
  • Instrument linking: The worksheet is linked to the specific instrument used for the run. Instrument calibration status is checked — if the instrument is overdue for calibration, the worksheet cannot be started. This is a system-enforced control: you cannot generate GxP results on an out-of-calibration instrument.
OQ test: Attempt to start a worksheet on an instrument whose calibration is expired. Verify system blocks with a clear error message. This is a critical GxP control test — instrument calibration status must gate result entry.

Result Entry, Review & Approval
The complete data flow from analyst entering a value through supervisor review to final QA approval — every step, every control, every e-signature.
▸ Click each stage in the approval chain
✍️ Analyst Result Entry

Results can be entered in three ways — all generate equivalent audit trail records:

  • Manual keyboard entry: Analyst types the value into the result field. System records: value, analyst ID, date/time.
  • Instrument import: Results transferred automatically from instrument system (Empower, Chromeleon) via file import or API. Audit trail records source as "instrument import" with file name and timestamp — distinguishable from manual entry (ALCOA Attributable).
  • Calculated from raw entry: Analyst enters raw data (e.g., peak area from instrument) and LBMS calculates the reportable result. Both raw entry and calculated result stored.
Critical: No result shopping. All instrument runs must be imported, not just the "best" run. If LabWare is configured for automatic import from the instrument, the system imports ALL runs. Analyst selecting only passing results to enter is a data integrity violation (ALCOA Original). This must be tested in OQ: perform 3 instrument runs, verify all 3 appear in LIMS.
⚠️ Spec Check — Immediate OOS/OOT Evaluation

The moment a result value is saved, LabWare evaluates it against the specification limits defined on the component. This is automatic, immediate, and cannot be bypassed.

  • OOS flag: Value outside spec limits → Result flagged OOS → Sample blocked from completion → OOS investigation auto-initiated
  • OOT flag: Value within spec but outside trend limits → Result flagged OOT → Notification to stability/QC lead → Not blocking but requires review
  • Visual indicator: OOS results highlighted in red on the result entry screen. Analyst cannot ignore it — it's the first thing visible upon save.
  • No manual override of OOS flag: The OOS flag is set by the system based on the spec. Only two ways to clear it: (1) OOS investigation that invalidates the result with documented scientific justification and QA approval, or (2) specification change (which requires change control and re-evaluation of all affected historical results).
🧮 LBMS Calculation Engine

When a result has a configured LBMS calculation formula, the formula fires on save — taking the entered raw value(s) and computing the reportable result.

Example — Assay calculation:

Raw entry: Peak Area = 845623 (analyst reads from instrument) LBMS formula: Assay % = (Peak Area Sample / Peak Area Standard) × (Standard Concentration / Sample Concentration) × (Dilution Factor) × 100 Calculated result: 99.4 % ← stored as the reportable value Raw entry: 845623 ← also stored permanently
Validation requirement (GAMP Cat 5): Every LBMS calculation formula must be independently verified before use. Write the formula in plain math notation in the FRS. Calculate the expected result manually using test data. OQ: enter the same test data into LabWare and confirm the system output matches the pre-calculated expected value. Any discrepancy = OQ failure = do not proceed to production.
👁️ Supervisor Review

After the analyst completes result entry for all tests on a sample, a QC Supervisor reviews the results before the sample can move to Completed status.

  • Supervisor reviews raw data attachments (instrument output files) against the reported results — verifying the numbers match what the instrument actually produced
  • Checks calculation accuracy (spot-check against manual calculation for complex assays)
  • Reviews any anomalies or analyst notes in the result record
  • Confirms all mandatory attachments are present (raw data file, calculation worksheet)
  • Supervisor e-sign (or initials) captured — separate event in the audit trail from analyst entry

Key point: Supervisor review is a QC-level control. It does not constitute QA approval. After supervisor review, the sample moves to Completed and awaits QA review.

✅ QA Approval — E-Signature & Record Lock

QA approval is the final gate before a sample's data can be used for batch release decisions. It requires a 21 CFR Part 11 compliant electronic signature.

  • E-signature event: QA user clicks Approve, system prompts for password re-entry. Signature meaning displayed: "I approve these results as meeting specification requirements and authorise their use for batch release." User confirms with password → signature captured.
  • What is checked before approval is allowed: All required tests completed, no open OOS investigations, all mandatory attachments present, supervisor review completed, no open data correction requests.
  • Record lock: Upon approval, the sample and all its result records are locked. No edits possible by any user (including admin) without a formal data correction workflow that generates its own audit trail.
  • Audit trail entry: E-signature event recorded: user ID, printed name, timestamp, signature meaning, and the state of the record at the time of signing.
Part 11 OQ test: Attempt to approve a sample where the QA user was also the analyst who entered results. System must block. Attempt with correct QA user → system prompts for password → correct password accepted → signature recorded. Verify the audit trail entry contains all required Part 11 elements.

OOS / OOT Handling
The most regulated workflow in your LIMS. How Out-of-Specification results are detected, investigated, and resolved — with full FDA guidance alignment.
▸ Click each phase of the OOS investigation
🚨 OOS Trigger — System Response

The moment an OOS result is saved, LabWare:

  • Sets OOS flag on the result record
  • Highlights the result in red on all screens
  • Blocks the sample from moving to Completed status
  • If configured: auto-creates an OOS Investigation record linked to the result
  • If configured: sends immediate notification to QC Supervisor and QA
  • Prevents the analyst from continuing work on the same test without acknowledging the OOS
FDA 2006 OOS Guidance: An OOS result must be reported to the laboratory supervisor immediately. LabWare's automatic notification satisfies this requirement electronically — but the supervisor must act, not just receive the notification. The acknowledgment action must be documented in the OOS investigation record.
🔬 Phase 1 — Laboratory Investigation

Phase 1 is a rapid, same-day laboratory investigation to look for an assignable laboratory cause — an error in the analytical process rather than a genuine product quality failure.

Phase 1 checklist (documented in the OOS record):

  • Was the correct standard used? Is the standard within its use-by date and properly stored?
  • Was the calculation correct? Check the formula, the dilution factors, the weight taken.
  • Was the instrument within calibration? Was instrument QC passing for this run?
  • Were there any analyst transcription errors between instrument output and LIMS entry?
  • Was the correct sample used? Could samples have been mixed up?
  • Was the sample prepared according to the method? Correct pH, temperature, extraction conditions?

Each item is checked and the finding documented. If a clear, documented, scientifically-justified lab error is found → proceed to invalidation. If no lab error found → mandatory escalation to Phase 2.

FDA critical rule: Phase 1 cannot be used to simply invalidate an OOS result because "we assume it's a lab error." A specific, documented, scientifically defensible assignable cause must be identified. "The analyst must have made an error" is not acceptable — you must identify what the error actually was.
✔️ OOS Invalidation (Lab Error Confirmed)

If Phase 1 identifies a clear, documented laboratory error:

  1. Document the specific assignable cause in the OOS record (e.g., "Standard solution prepared at wrong concentration — calculation shows 10x dilution error consistent with the observed OOS value")
  2. QA reviews and approves the invalidation — requires QA e-signature with meaning "I approve the invalidation of this result based on the documented laboratory error"
  3. The original OOS result is marked "Invalidated" — it remains in the system permanently with its full audit trail. It is never deleted.
  4. A retest is authorised. The retest is a fresh, complete analytical sequence — not a re-read of the same instrument run.
  5. Retest result entered. If passing → sample can proceed to Completed. If failing again → Phase 2 mandatory.
ALCOA Original: The invalidated OOS result is preserved permanently. You can never delete it. An inspector can always see that an OOS result occurred and trace what happened to it. This is the correct behaviour — the audit trail tells the complete story.
🔴 Phase 2 — Full OOS Investigation

Phase 2 is a formal, expanded investigation that goes beyond the laboratory. It must be completed within a defined timeframe (typically 30 days, per your SOP).

Phase 2 scope:

  • Full retest of original sample: Complete retesting from scratch — new sample preparation, new standard, different analyst, same instrument type. Minimum number of retests defined in your OOS SOP (typically n=2 or n=3).
  • Retain sample testing: Test the retained reference sample from the same batch.
  • Manufacturing review: Examine batch manufacturing records for the production period — any deviations, equipment issues, raw material anomalies, process parameter excursions.
  • Raw material review: Confirm CoA of all raw materials used in the batch.
  • Environmental data: For temperature-sensitive products — review environmental monitoring data for production and storage periods.
  • Statistical evaluation: Are the retest results consistent with each other? A wide spread in retest results suggests a lab variability issue. Consistently low results confirm a product failure.

Phase 2 outcomes:

  • Confirmed OOS (product failure): Batch rejected. Formal disposition (Reject/Quarantine). Regulatory notification if required. CAPA raised.
  • Inconclusive (additional testing): More retests with statistical justification. Cannot retest indefinitely — a pre-defined maximum retest number must be stated in your SOP.
✅ OOS Closure & Batch Disposition

The OOS investigation closes when all Phase 2 activities are complete and a documented conclusion is reached. The closure must be QA-signed.

  • Batch Fail: Batch disposition = Reject. QA closes OOS with a reject decision. Manufacturing is notified. Regulatory reporting assessed (is this a significant quality event requiring health authority notification?). CAPA raised if the root cause is systemic.
  • OOS Invalidated (Phase 2 confirms lab error): Rare outcome from Phase 2. Requires very strong evidence. The original OOS result must be explicitly documented as laboratory-attributable in the Phase 2 conclusion with QA and ideally QP/RP sign-off.
Averaging rule (FDA 2006 Guidance): You cannot average an OOS result with passing retest results to calculate a "passing average." Each result must stand on its own merits. If 1 of 3 retests is OOS, investigate the outlier — do not average it away. This is one of the most common OOS-related warning letter citations.

Batch Release & Certificate of Analysis
The final quality gate before product leaves the facility. How LIMS supports the batch release decision and generates the CoA that travels with the product.
🏭 Batch Release Workflow

Batch release in LabWare is the process of confirming that all required analytical testing for a batch has been completed, all results are within specification, and QA has formally approved the batch for distribution.

Prerequisites for batch release (system-enforced gates):

  • All required tests for the product (per its test plan) have a sample in Approved status
  • No open OOS investigations linked to this batch
  • Stability T=0 samples logged (if applicable)
  • All sample records for the batch are in Approved status (no open Completed or In Progress samples)

CoA Generation:

  • QA initiates CoA generation from the batch record. LabWare aggregates all approved test results for the batch into a Crystal Reports template.
  • CoA includes: product name, batch number, manufacturing date, expiry date, test name, specification, result, pass/fail for every tested parameter
  • QA reviews the CoA on-screen. Any discrepancy between the CoA and the LIMS results = data integrity flag = do not release
  • QA applies e-signature to the CoA with meaning "I certify this batch meets all release specifications." Signature is embedded in the CoA document and in the LIMS audit trail.
  • Approved CoA is generated as a PDF, stored in LabWare as an attachment to the batch record, and transmitted to the customer/distribution system.
CoA integrity rule: A CoA generated from LabWare must exactly reflect the approved LIMS data. The CoA PDF is locked (digitally signed) upon generation — it cannot be modified after QA signature. Any reissued CoA must go through a formal reissue workflow with documented reason and QA approval. Manually edited CoAs (Word, PDF editor) are a critical data integrity violation.

Stability Management
How LabWare manages long-running stability studies — protocol setup, automated pull scheduling, OOT detection, and shelf-life calculations.
📅 Stability Program Architecture in LabWare

Stability studies in LabWare are managed through a Stability Protocol → Study → Schedule → Pull → Sample hierarchy:

  1. Stability Protocol — Master configuration: storage conditions (25°C/60%RH = Long Term, 40°C/75%RH = Accelerated, 30°C/65%RH = Intermediate), timepoints (T=0, T=1mo, T=3mo, T=6mo, T=9mo, T=12mo, T=18mo, T=24mo, T=36mo, T=48mo, T=60mo), test requirements at each timepoint, acceptance criteria.
  2. Stability Study — Instance of a protocol for a specific batch. Created when T=0 samples are logged. Contains all the scheduled pulls for that batch across all conditions.
  3. Automated Pull Scheduling — LabWare calculates the pull date for every timepoint from the T=0 date. Generates alerts 30/14/7 days before each pull is due. If a pull is overdue (beyond the ICH-defined tolerance window), it flags as a deviation.
  4. Pull tolerance windows — ICH Q1A defines allowable windows for each timepoint (e.g., T=3mo: Day 90 ± 7 days). LabWare enforces these windows. A sample pulled outside the tolerance window auto-creates a deviation record.
  5. OOT Detection — Results within specification but showing a downward trend are flagged as Out of Trend. LabWare uses statistical methods (linear regression trend analysis or designed trend tests per ICH Q1E) to evaluate stability trends over time.
  6. Shelf-life projection — From trend analysis, LabWare can project when a parameter will reach the specification limit — enabling proactive label claim review before the expiry date becomes an issue.
Regulatory basis: ICH Q1A (Stability Testing), ICH Q1E (Evaluation of Stability Data), FDA 21 CFR 211.166 (Stability Testing). Your LabWare stability configuration must align to these guidelines. The configuration (protocol timepoints, tolerance windows, OOT thresholds) is validated configuration — documented in the Configuration Spec and tested in OQ.

Instrument Integration
Bidirectional LIMS-instrument communication — how worklists go to instruments, how results come back, and how you validate each interface.
▸ Click each integration pattern
🧪 HPLC Integration (Empower / Chromeleon)

HPLC is typically the most complex instrument integration — high data volume, multi-component results, and strict data integrity requirements.

Integration flow (bidirectional):

  • LIMS → HPLC (worklist push): LabWare exports a sample worklist (sample IDs, methods, vial positions) to the CDS (Chromeleon or Empower). Analyst imports the worklist into the CDS, eliminating manual sample ID transcription.
  • HPLC → LIMS (results pull): After the run, the CDS exports results (peak areas, retention times, calculated assay values) as a structured file. LabWare LBMS parser reads the file, maps each result row to the correct LIMS sample/test/component by sample ID, and imports all results in a single batch.
  • Raw data linkage: The original CDS data file (or a PDF rendition) is attached to the LIMS result record as supporting raw data. Auditors can trace from the LIMS result back to the original chromatogram.
Data integrity — all runs imported: The LBMS import script must import ALL results from the CDS file — not just passing ones. If the import script has logic to skip OOS results, that is a data integrity violation. OQ test: run samples that produce an OOS result, import the file, verify the OOS result appears in LIMS with the OOS flag set.
⚖️ Balance / LabX Integration

Balance integration (Mettler Toledo LabX, Sartorius LabX, or direct serial connection) captures weighing data directly into LIMS — eliminating manual transcription of weights.

  • Task-based workflow: LIMS sends a weighing task to the balance (via LabX middleware). Analyst places the sample on the balance. Balance sends the weight back to LIMS, which records it against the correct sample and test.
  • WeightID uniqueness: Each weighing task must carry a unique WeightID that links the weight measurement to the specific LIMS sample record. The WeightID uniqueness constraint must be enforced at both application and database layer. (This is the exact root cause of the 2024 LabX incident at Gilead — missing uniqueness constraint allowed duplicate WeightIDs to be created, creating an unattributable measurement.)
  • Audit trail: The balance records the operator, date/time, and weight value. LabX passes this to LIMS. The LIMS result record shows source = "balance import" with instrument ID and timestamp.
💊 Dissolution System Integration

Dissolution testing typically involves UV/Vis spectrophotometry or HPLC for quantification, with the dissolution apparatus controller managing the physical test.

  • Dissolution results include both the dissolution profile (% dissolved at each timepoint) and a final Q-value (pass/fail vs. specification)
  • Integration typically exports a results file from the dissolution software (Agilent SOFTmax, Hanson Elite, Sotax AT) to a watched folder, which LabWare's import engine picks up
  • For multi-vessel runs (e.g., 6 or 12 vessels per batch), the import maps each vessel position to the correct test component
  • LBMS calculates whether the mean % dissolved at each specified timepoint meets the Q criterion
🦠 Microbiology System Integration

Microbiology LIMS integration is more varied — ranging from manual colony count entry to automated rapid microbiological systems.

  • Colony count entry: Analyst reads physical plates and enters counts manually. LBMS calculates CFU/g or CFU/mL from the count, dilution factor, and volume plated. OOS evaluation: result exceeds the microbial limit specification.
  • Rapid micro systems (e.g., bioMérieux BacT/Alert, Biolog, MALDI-TOF): System exports identification or enumeration results as structured files → imported to LIMS via parser script.
  • Environmental monitoring: LIMS manages EM sample schedules, sampling location tracking, and trend analysis across all monitoring points (ISO rooms, critical surfaces, personnel). Alert levels and Action limits evaluated per the EM program specification.
🔄 Middleware Options

For complex multi-instrument environments, a middleware layer sits between instruments and LIMS:

  • NuGenesis (Waters): Scientific Data Management System — captures raw data from multiple CDS and instrument systems, provides a managed repository, and integrates with LabWare via a validated connector
  • LabX (Mettler Toledo): Balance/titration middleware — manages weighing tasks and interfaces with LabWare via the task API
  • Custom middleware: Python/Java services built in-house to translate proprietary instrument formats to LabWare's import format. These are GAMP Cat 5 and require full SDLC validation.

CSV & Validation Strategy
How to validate LabWare end-to-end — GAMP classification, critical OQ test cases, Part 11 compliance checklist, and what an FDA inspector will ask you to show them.
📋 LabWare Validation — Complete Scope

GAMP Classification:

  • Standard LabWare platform (uncustomised) = GAMP Cat 4 (configured COTS)
  • Any LBMS script written by your team = GAMP Cat 5 (custom software)
  • Infrastructure (Oracle DB, WebLogic, OS) = GAMP Cat 1 (infrastructure)

Critical OQ test cases — must pass before go-live:

  • Sample login with auto-test assignment for all product types
  • Result entry — manual and instrument import
  • OOS flag fires on correct value range; does not fire within spec
  • OOS blocks sample completion — cannot be bypassed
  • E-signature — correct user, password re-entry, meaning captured
  • Self-approval blocked — analyst cannot approve own results
  • Audit trail — all field changes captured with old/new value
  • LBMS calculations — every formula with pre-defined test cases
  • Status-based locking — Approved records cannot be edited
  • Instrument interface — all fields map correctly, all runs imported
  • CoA generation — every field matches LIMS approved data
  • Roles — QC Analyst cannot approve, Read-Only cannot edit
  • Uniqueness constraints on all identifier fields (WeightID, SampleID, Barcode)
  • Inactivity timeout forces re-login
  • Instrument calibration gate — expired calibration blocks worksheet
  • Stability pull scheduling — correct dates, tolerance window enforcement
  • Backup restore — full system recoverable from backup
  • ERP integration — correct data flows both directions
  • Stability OOT detection fires at correct threshold
  • Reports — audit trail, CoA, OOS summary match source data
What the FDA inspector will ask to see: (1) Your validation package — IQ/OQ/PQ + RTM with 100% coverage. (2) The audit trail for a specific result — show original entry, any corrections, approvals. (3) Your OOS procedure and a real OOS case from the past year — show the investigation record, the retest, the outcome. (4) Your access control records — who has QA Approver role, when were they added. (5) Your LBMS script inventory — can you produce a list of every script running in production with its validation evidence.

Upgrades, Maintenance & Periodic Review
Keeping the validated state through the full system lifetime — patches, major upgrades, periodic reviews, and long-term maintenance strategy.
🔄 LabWare Upgrade Strategy

LabWare (on-premise) gives you full control over your upgrade schedule — and full responsibility. Unlike SaaS, patches don't auto-apply. This is both an advantage (no surprise changes to your validated system) and a risk (security patches must be proactively managed).

Major version upgrade (e.g., v7.0 → v7.5 or v7 → v8):

  1. Gap analysis between versions — what changed in the platform that affects your configured GxP functions
  2. Clone PROD to upgrade test environment
  3. Apply upgrade in test environment, execute full regression suite (Tier 3)
  4. Special focus: LBMS scripts must be re-tested — syntax changes between major versions can silently alter behaviour
  5. Crystal Reports re-validated — report output must be compared field-by-field before and after upgrade
  6. All instrument interfaces re-tested with live test data
  7. UAT by business users before go-live sign-off
  8. Scheduled maintenance window for production upgrade, with tested rollback plan

Periodic Review checklist:

  • Admin audit trail reviewed — all production admin actions correspond to approved change controls
  • LBMS script inventory updated and all scripts tested
  • User access list reviewed — stale accounts disabled, roles appropriate
  • Open deviations and OOS records reviewed for trends
  • Backup verification records reviewed — monthly restores documented
  • SOPs current — all reference correct system version
  • Training records complete for all current users
Director-level insight: "The labs that struggle with LabWare upgrades are the ones with the largest LBMS script debt — hundreds of undocumented scripts built over years with no version control. Our first priority should be a full script inventory and documentation exercise before the next major upgrade. That work protects the upgrade timeline and the validated state."
