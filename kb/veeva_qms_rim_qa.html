Architecture & Configuration
Vault structure, object model, security, and admin fundamentals
Q01Admin
What is the Vault object model and how does it differ from a relational database? How do you design custom objects in QMS?
▼
Answer

Vault uses a metadata-driven object model. Every piece of data — whether a document, a CAPA record, or a training assignment — is an object with fields, lifecycle states, and relationships. This differs from a relational DB where you design tables and joins manually.

Core concepts:

  • Object Types — Subtypes of a base object (e.g., "CAPA" and "Deviation" can both be types of "Quality Event"). Each type can have different fields, lifecycles, and page layouts.
  • Object Fields — Standard (system-managed) vs. custom fields. Field types: text, picklist, date, lookup, object reference, multi-value picklist.
  • Object Relationships — Parent-child (cascade delete behavior), lookup (reference without ownership), many-to-many via junction objects.
  • Page Layouts — Control which fields appear for which object type and role. One object type can have multiple layouts for different user roles.

Design best practice: Before creating a custom object, map out: What fields are required? What lifecycle states? Who can create/edit/delete? What relationships to existing objects? Document this in a Configuration Specification before building in Admin.

Audit consideration: Every custom object, field, and picklist value created in a validated vault must be tracked in a Configuration Specification document and tested in OQ/PQ. Configuration changes are change-controlled.
Q02Admin
How does Vault security model work — Vault Access, Object Permissions, and Document Roles? What's the difference and how do they interact?
▼
Answer

Vault uses a layered security model with three distinct layers that must all be satisfied for a user to do anything:

  1. License Type — Full User, External User, or Read-Only User. Controls what module features are available at all.
  2. Security Profile — Grants access to the Vault application itself and to Admin features. Controls UI tabs and admin menu access.
  3. Permission Sets — Control CRUD (Create/Read/Update/Delete) on Object Types and specific field-level permissions. Assigned to users or groups.
  4. Document Roles (for Documents) — Viewer, Editor, Owner, Coordinator. Assigned at the document or binder level or inherited via sharing rules. Determines who can edit, review, approve a document.
  5. Sharing Rules & Dynamic Access Control — Automatically assign document roles based on field values (e.g., "All documents where Department = QA get the QA Reviewer role assigned to the QA group").

Key interaction: A user needs the right Permission Set (to see the object) AND the right Document Role (to act on a specific document). Having one without the other results in no access.

Validation implication: Your OQ must test each role combination — including negative tests proving that Role A cannot perform Role B's actions. The Permission Set matrix must be documented in your Configuration Spec and tested against your URS access requirements.
Q03Both
What environments exist in Veeva Vault and how do you manage the promotion of configuration changes from Dev to Production?
▼
Answer

Veeva Vault provides multiple vault instances — typically referred to as environments:

  • Development (DEV) Vault — Where new configurations are built and unit-tested. Sandbox. Not GxP.
  • QA / Validation Vault — Where formal validation testing (OQ/PQ/UAT) is executed. This vault must mirror Production configuration at the point of testing.
  • Production (PROD) Vault — Live GxP system. Changes only enter after approved change control and successful validation testing.

Configuration Migration: Veeva provides the Vault Loader and Configuration Migration (now largely via Vault's built-in deployment packages / VPK files). An admin exports a deployment package from DEV and imports it to QA/Prod. The package is reviewed and approved as part of the Change Control.

Val Lead angle: The migration itself is a validated step. Your OQ should include a test that verifies the configuration deployed to QA matches the Configuration Spec exactly. Post-deployment to PROD, a configuration verification step should confirm settings are as expected — this becomes evidence in the Change Control closure.
Q04Admin
Explain Vault's Lifecycle and Workflow engine. What is the difference between a Lifecycle State and a Workflow Task?
▼
Answer

Lifecycle is the overall state machine for a document or object. It defines the possible states (e.g., Draft → In Review → Approved → Superseded → Obsolete) and the allowed transitions between them.

State Actions & Entry/Exit Actions — You can configure automatic actions when entering or leaving a state: auto-assign tasks, send notifications, trigger field updates, lock/unlock fields.

Workflow is a structured process assigned to a document/object to orchestrate human tasks. A workflow has Tasks (assigned to specific users or groups) with actions: Approve, Reject, Complete.

Key difference:

  • Lifecycle defines WHERE the document is (its state). It's always active.
  • Workflow defines WHO needs to do WHAT to move it. It's triggered on demand or by a lifecycle entry action.
  • A lifecycle transition can be conditional — it only moves to the next state after a workflow is completed (e.g., document cannot move to Approved until the Approval Workflow task is completed by the designated approver).
Admin tip: Workflows can be multi-step with parallel and sequential tasks. Use workflow controls to define what happens on rejection — does it return to Draft, notify the initiator, or route to a manager? This logic must be explicitly configured and tested in OQ.

Q05Both
Walk through the full lifecycle of a Deviation record in Vault QMS — from initiation to closure. What states, workflows, and validations are typically involved?
▼
Answer

A typical Deviation lifecycle in Vault QMS flows through these states:

  1. Draft — Initiator describes the event, assigns affected products/processes, selects deviation type (Planned/Unplanned). Required fields enforced before state transition.
  2. Under Investigation — Root cause analysis conducted. QA assigned as owner. Entry triggers workflow task to QA reviewer. Immediate containment actions documented.
  3. Pending QA Review — QA reviews investigation, root cause, and impact assessment. Workflow task: Approve → moves forward; Reject → returns to Investigation with comments.
  4. CAPA Required (conditional) — If risk score meets threshold (configurable via formula fields), system auto-creates a linked CAPA record. Relationship established (Deviation is parent).
  5. Pending Closure — All required fields completed. CAPA (if required) must be in Approved or Implemented state. Entry criteria enforced via lifecycle entry actions.
  6. Closed — Final QA signature (e-signature). Record locked. Audit trail complete.
Validation scope: Each state transition is an OQ test case. Negative tests: attempt to close a deviation with open CAPA — system must block. E-signature on closure must be tested for Part 11 compliance. Auto-CAPA creation must be tested with values both above and below the threshold.
Q06Admin
How do you configure automated notifications and escalations in Vault QMS? Give examples of overdue CAPA escalation logic.
▼
Answer

Vault uses Workflow Notifications and Object Event Triggers (including time-based triggers via Scheduled Actions) for this.

Overdue CAPA Escalation Example:

Object: CAPA Field: Due Date (date field) Field: Status (picklist: Open, In Progress, Completed, Overdue) Field: Owner (user lookup) Field: Manager (user lookup, set from Owner's profile) Scheduled Action (runs daily): Condition: Due Date < TODAY() AND Status != "Completed" Action 1: Update Status field → "Overdue" Action 2: Send Email Notification → Owner + Manager Action 3: If Due Date < TODAY() - 7 days → Notify QA Director

Workflow task reminders — Within a workflow, each task can have a reminder notification (e.g., "Reminder: Task due in 3 days") and an overdue notification (e.g., "Task is past due — notify task owner + their manager").

OQ test: You must test each notification fires correctly. Create a CAPA, set Due Date to yesterday, run the scheduled action (or advance date in a sandbox that allows it), verify notification sent to correct recipients, verify Status updated to Overdue.
Q07Val Lead
How do you validate a QMS workflow change mid-system lifecycle? A business user requests a new mandatory field on CAPA — how do you handle this end-to-end?
▼
Answer

This is a classic post-go-live enhancement through the change control process:

  1. Enhancement Request / Change Request opened — Business user submits ER describing: which object (CAPA), what field (e.g., "Risk Level" picklist), why (regulatory gap identified), business justification.
  2. Impact Assessment — Val Lead assesses: Does this affect a validated GxP function? YES — CAPA is GxP-critical. Does it affect existing open records? YES — data migration consideration. Does it affect any integrations? Check if CAPA data feeds any downstream system.
  3. Requirements Update — FRS and Configuration Spec updated to include new field definition, values, mandatory conditions.
  4. RTM Update — New RTM row added: ER-001 → FRS-045 → OQ test case TBD.
  5. Configuration in DEV — Admin creates field, sets required condition (e.g., required on transition to "Pending Closure"), adds to page layout for all CAPA types.
  6. OQ Re-testing in QA Vault — Write targeted OQ script: (a) Create CAPA, attempt closure without Risk Level — system must block. (b) Create CAPA, enter Risk Level, complete closure — passes. (c) Verify field appears in audit trail. (d) Verify field label and picklist values match FRS.
  7. UAT in Staging — CAPA owner business user confirms the field makes sense in their workflow.
  8. Production Deployment + Change Control Closure — Config migrated to PROD. Post-deployment verification step: confirm field exists, is mandatory on correct transition. Change control closed with test evidence attached.
Migration note: Existing open CAPA records may not have the new field populated. Decide: Is the field conditionally required (only on new records) or will you need a data remediation exercise for existing records? This must be risk-assessed and documented.

Q08Admin
How does Vault handle document versioning — major vs. minor versions? How do you configure version numbering and when does each increment?
▼
Answer

Vault uses a Major.Minor version numbering scheme configurable per document type:

  • Major version increments on Approved/Published state entry. Configurable as numeric (1, 2, 3) or alphabetic (A, B, C). Indicates a substantive content change that required full review and approval.
  • Minor version increments during Draft/Review stages within a major version (0.1, 0.2, 0.3). Tracks in-progress revisions before approval. Draft versions are not shown to most users — only owners and reviewers.

Steady-state versioning:

  • SOP starts as 0.1 (Draft), goes through review cycles (0.2, 0.3), approved → becomes 1.0.
  • Revision initiated → creates 1.1 (Draft). Revised, approved → becomes 2.0. Previous 1.0 is automatically Superseded.

Configuration: In Admin → Document Types → Version Scheme. You also configure whether the previous approved version remains viewable (it should — for historical audit trail) or is archived.

Common OQ test: Verify that when v2.0 is approved, v1.0 automatically moves to Superseded state. Verify that only the latest Approved version is visible to Viewer-role users. Verify minor versions are not visible to non-owners.
Q09Both
How do you configure and manage Periodic Review for controlled documents in Vault QMS? What happens when a periodic review is overdue?
▼
Answer

Periodic Review in Vault is driven by a calculated field on the document:

  • Next Review Date — Formula field calculated as: Approval Date + Review Period (e.g., Approval Date + 730 days for 2-year review cycle). Configurable per Document Type.
  • Periodic Review Status — Picklist: Current, Due Soon, Overdue. Updated by a scheduled action comparing Next Review Date to TODAY().

Workflow trigger: When a document reaches "Due Soon" (e.g., 30 days before due), a workflow task is auto-created and assigned to the document owner: "Initiate Periodic Review." Owner reviews, decides: No Change Needed (re-approves to reset Next Review Date) or Initiate Revision (starts full change workflow).

Overdue behavior: If task is not completed by Next Review Date, document status flags as Overdue. If your policy requires it, lifecycle can be configured to move the document to a "Requires Immediate Review" state, restricting it from being used in training assignments until resolved.

Regulatory risk: Having SOPs in Overdue status at the time of an FDA inspection is a significant finding. Your periodic review SOP must define the frequency, responsible parties, and what constitutes acceptable completion. The Vault report showing overdue documents is a key inspection-ready artifact.

Deviations, CAPAs & Change Controls

Q10Both
How do you model the relationship between a Deviation, a CAPA, and a Change Control in Vault? Can one CAPA be linked to multiple deviations? How is this configured?
▼
Answer

These three objects are related through Object Relationships in Vault. The standard configuration:

  • Deviation → CAPA: Parent-Child. A Deviation can spawn one or more CAPAs. The CAPA has a "Source Deviation" lookup field. Multiple deviations can share one CAPA if root cause analysis identifies a common systemic issue — this is a many-to-many relationship, modeled via an intermediary relationship object or a multi-value object reference field.
  • CAPA → Change Control: Lookup relationship. A CAPA's corrective action may require a formal Change Control to update an SOP, retrain staff, or modify a process. The CC record has a "Linked CAPA" reference field.
  • Deviation → Change Control: Direct lookup for planned deviations that themselves are change events.

Closure dependency: Configure a lifecycle entry criterion on CAPA closure: "All linked Change Controls must be in state = Implemented." This is done via object lifecycle entry actions with a related record count validation.

Reports matter: Auditors request "Show me all open deviations with their linked CAPAs and CC status." Your Vault report must pull across these relationships. Build and validate a "QE Linkage Report" as part of your validation package.
Q11Val Lead
A QC lab migrates 3,000 historical deviation records from a legacy LIMS into Vault QMS. What validation activities are required for this data migration?
▼
Answer

Data migration of GxP records is a validated activity with its own protocol. Steps:

  1. Data Migration Plan — Document scope (which records, which date range), source system extract method, field mapping (Source Field → Vault Field), data transformation rules (e.g., legacy status codes mapped to Vault picklist values), handling of incomplete legacy records.
  2. Field Mapping Specification — Every source field mapped to its Vault destination. Unmappable fields documented with disposition (discard, notes field, separate attachment).
  3. Extraction & Transformation (ETL) — Extract from legacy, apply transformations, load to Vault via Vault Loader (CSV bulk import) or API. Migration runs in DEV/QA vault first.
  4. Migration Verification Protocol (OQ-level) — Statistically sampled or 100% verification depending on risk:
    • Record count: Source count = Vault count
    • Field-level spot checks (randomized sample): 20–100 records
    • Data integrity: Key identifiers match (deviation ID, date, owner, status)
    • Relationships intact: CAPA links preserved
    • Attachments migrated: PDF files attached to correct records
  5. Cutover Plan — On go-live day: legacy system made read-only → final delta migration → verification run → sign-off to proceed to production load.
Audit trail note: Migrated records will show the migration service account as the "Created By" user with the migration date — not the original author. You MUST document this in a Migration Addendum to your validation package, and store the original records (or a certified extract) as evidence. Never alter migrated records to fake original authorship.

Training Management

Q12Admin
How does Vault QMS Training module work? How are training assignments auto-generated when a new SOP version is approved?
▼
Answer

Vault Training is built on Training Assignments generated automatically based on:

  • Training Requirements (Curricula) — A set of documents that a specific job role must be trained on. Configured per Document Type + Job Role combination. When a user is assigned a Job Role, they're enrolled in the corresponding curriculum.
  • Auto-Assignment on Approval — When a document moves to Approved state, Vault checks: "Which Job Roles have this document type in their curriculum?" It then auto-creates a Training Assignment for every user in those roles. The assignment has a due date calculated from the document Approval Date + Grace Period (configurable).

Training Assignment workflow:

  1. User receives notification of new training assignment.
  2. User reads the document (system can track "Read and Understood" acknowledgment — user clicks "Complete Training" after reading the linked document).
  3. Optional: quiz/assessment configured (external LMS link, or built-in acknowledgment).
  4. Training Assignment moves to "Complete" state with user's e-signature and timestamp.
Key OQ test: Approve a test SOP version → verify training assignments auto-created for all users in the configured roles → verify no assignments created for users NOT in those roles → verify due date calculated correctly → user completes assignment → verify e-signature captured → verify assignment record reflects correct document version and approval date.
Q13Val Lead
A new employee starts but is given system access before completing their training curriculum. What are the GxP implications, and how should Vault be configured to prevent this?
▼
Answer

GxP implication: An untrained user accessing a GxP system and performing regulated tasks is a critical data integrity violation. Any data they entered could be challenged in an inspection. Training records are a regulatory prerequisite to system access, not a post-access formality.

Vault configuration to enforce this:

  • Option 1 — Process control: Access provisioning SOP requires training completion before IT creates the Vault account. Training completion proof submitted with access request.
  • Option 2 — Vault enforcement: User account created but in Inactive status. Activation is a manual step by the system admin, triggered only when the onboarding training workflow is complete. The workflow sends a notification to the admin to activate the account.
  • Option 3 — Permission-based restriction: New users are provisioned with a "Trainee" Permission Set that allows read-only access. Upon training completion, the admin upgrades their Permission Set to their full operational role.
Val Lead action: This requirement must be captured in the URS ("The system shall not permit users to perform GxP functions prior to completion of required training") and tested in OQ. The chosen mechanism — process-based or system-enforced — must be documented with justification in the Validation Plan.

Audits, Audit Trails & Inspections
Audit management module, electronic audit trail, and inspection readiness
Q14Both
How does Vault's audit trail work technically? What events are captured, who can access it, and how do you demonstrate audit trail integrity to an FDA inspector?
▼
Answer

Vault maintains a system-generated audit trail that cannot be disabled by any admin or user. It records:

  • Every document and object create, edit, delete
  • Every field value change (old value → new value)
  • Every lifecycle state transition
  • Every e-signature event (user, meaning, date/time)
  • Every login/logout (in the user access audit log)
  • Every document download or view (document access log)

Access: The audit trail is viewable by System Admins via Admin → Audit Trail. Individual record audit trails are visible to users with the appropriate permission on the document/object itself. Standard users cannot view the system-wide audit log — only record-level history on records they have access to.

Demonstrating integrity to FDA:

  1. Show that the audit trail is ON by default and cannot be toggled off (this is documented in Veeva's 21 CFR Part 11 whitepaper — which should be in your validation package).
  2. Show a live record where a field was edited — demonstrate the audit trail shows original value, new value, who made the change, and when.
  3. Show that audit trail entries have no Edit button — they are read-only.
  4. Provide your audit trail review SOP showing you periodically review the audit trail for anomalies.
Vendor SOC 2 Report: Veeva provides a SOC 2 Type II report attesting to its infrastructure controls. This supplements your validation package and is often requested by auditors to verify Veeva's own controls over the platform infrastructure.

Q15Val Lead
Vault QMS is a SaaS platform. How does your validation strategy differ from an on-premise system? What does Veeva provide and what do you still own?
▼
Answer

Vault QMS is GAMP Category 4 (configured COTS). Being SaaS changes the validation boundary but not the obligation:

What Veeva owns and provides:

  • IQ-equivalent evidence — Infrastructure qualification, datacenter certifications (SOC 2, ISO 27001), disaster recovery, backup.
  • Platform OQ — Veeva provides a "Test Factory" or their own validation documentation for core platform features (standard workflow engine, audit trail, e-sig). You can leverage these under a Vendor-Supplied Testing (VST) strategy with documented vendor assessment.
  • Release Notes — Veeva publishes detailed release notes for every sandbox and production release.

What YOU still own:

  • Configuration OQ/PQ — Everything you configured (custom objects, workflows, lifecycles, notifications, formulas) must be tested by you. Veeva does not test your configuration.
  • URS, FRS, RTM — Your business requirements, fully your responsibility.
  • SOPs and Training — Process-level controls, your responsibility.
  • Vendor Assessment — Formal assessment of Veeva as a GxP software supplier. Review their Quality Agreement, audit rights, SLA, and change notification processes.
Critical SaaS risk: Veeva pushes updates (typically 3 major releases per year) to your PRODUCTION vault on a fixed schedule. You must assess each release for GxP impact, test in sandbox, and document regression testing BEFORE the release reaches production. This is an ongoing validation obligation — not a one-time activity.
Q16Val Lead
How do you structure regression testing for a Veeva Vault QMS release (e.g., the Spring '25 release)? What is your process from release notes to production go-live?
▼
Answer
  1. Release Notes Review (4–6 weeks before prod release) — Review Veeva release notes line by line. Categorize each change: Platform core change, Feature enhancement, Bug fix, Security patch. Identify which items touch your configured GxP functions.
  2. Impact Assessment Document — For each impacted area, rate risk: High (re-test required), Medium (targeted regression), Low (no re-test, document rationale). Total up regression test scope.
  3. Sandbox Preview Testing (3–4 weeks before prod) — Veeva deploys the release to your sandbox vault first. Execute regression test scripts against sandbox. Log defects if any (raise Veeva support tickets for platform bugs).
  4. Regression Test Evidence — Pass/fail results documented. Any open issues either resolved, deferred with documented risk acceptance, or escalated to delay production release (via Veeva's controlled rollout options).
  5. Release Memo / Release Impact Assessment Report — Summary document signed by System Owner + QA: "Release X has been assessed. Regression testing completed. No critical issues. Approved for production."
  6. Production Release — Veeva applies the release. Post-release smoke testing: verify 5–10 critical workflows are functional. Documented in a post-deployment verification checklist.
Efficiency tip: Maintain a core regression script library — a set of ~20–30 test cases covering your highest-risk configured functions. Run these on every release. Reserve deep testing for release-specific impacted areas. This prevents "test everything every time" fatigue while maintaining coverage.

RIM Architecture & Modules
Vault RIM structure, core modules, and how they interconnect
Q17Admin
What are the four core modules of Vault RIM and how do they relate to each other? Walk through the data flow from a clinical trial to a regulatory submission.
▼
Answer

Vault RIM consists of four integrated modules:

  • RIM Submissions — Central hub. Manages the submission object hierarchy: Application → Submission → Submission Document. Tracks what is being submitted, to which agency, in which market, and in what format (eCTD, paper, regional).
  • RIM Submissions Archive — Stores the final published eDossier (eCTD sequences). The authoritative record of what was actually sent to regulatory agencies. Linked to Publishing outputs.
  • RIM Registrations — Manages the registration status of products in each market. Tracks approvals, conditions of approval, lifecycle events (renewals, variations, withdrawals). Links back to the Submissions that achieved each registration.
  • RIM Publishing — Manages the assembly and publishing of eDossier packages (eCTD). Takes source documents, applies STF (Submission Table of Figures) metadata, and generates eCTD sequences for electronic submission to agencies like FDA (ESG), EMA (CESP), or Health Canada.

Data flow for an NDA:

Clinical/CMC documents (in Vault or external) → Authored, reviewed, approved in QMS/Vault Docs → Linked to Submission (NDA) via Submission Document records in RIM Submissions → Publishing team assembles eCTD sequence in RIM Publishing → Published eCTD archived in RIM Submissions Archive → Submitted to FDA via ESG → Approval tracked in RIM Registrations (NDA Approval, conditions, labeling) → Post-approval changes tracked as Variations/Supplements
Q18Admin
What is the RIM object hierarchy — Application, Submission, Activity? How do you configure this structure for a global product with multiple markets?
▼
Answer

The RIM hierarchy in Vault is:

  • Product / Registered Product — Top level. Represents the drug product or medical device. Has fields for product name, active substance, therapeutic area, development stage.
  • Application — Regulatory application per country and application type. E.g., NDA 123456 (US), MAA EU-2024-001 (EU), NDS 12345 (Canada). Each application belongs to one Product and one Country.
  • Activity — An action within an application lifecycle. E.g., Initial Submission, Annual Report, Prior Approval Supplement, Labeling Update. Activities track the regulatory event lifecycle.
  • Submission — The actual submission package sent to an agency. Linked to an Activity. Contains Submission Document records that link to the actual documents.
  • Submission Document — Junction object linking a Submission to a Vault Document. Has metadata: CTD section (e.g., 3.2.S.1), document version, submission-specific title.

For a global product: One Product → Multiple Applications (one per country) → Each Application has its own Activity/Submission hierarchy. The same source documents can be linked to multiple submissions via separate Submission Document records — enforcing single source of truth without duplicating files.

Admin consideration: Picklist values for Country, Application Type, Activity Type, CTD Section must be configured in Admin and kept current with regulatory guidance changes. These are reference data — changes to picklist values require a change control.
RIM
Submissions Management
Submission planning, document management, agency communications, and sequence management
Q19Both
How does Vault RIM manage the link between source documents and regulatory submissions? What happens to a submission when a source document is revised?
▼
Answer

Vault RIM maintains a version-specific link between submissions and documents. The Submission Document record references a specific version of the source document — not just the document itself.

When a source document is revised:

  1. The document moves through its authoring/review/approval lifecycle and a new major version is approved (e.g., v1.0 → v2.0).
  2. The previous Submission Document record (pointing to v1.0) remains intact — preserving the historical record of what was actually submitted.
  3. The Regulatory Affairs team assesses: Does this revision require a new submission (e.g., a CMC change that triggers a CBE-30 or PAS)? If yes, a new Activity and Submission are created, and a new Submission Document record is created pointing to v2.0.
  4. Vault's Document Tracker (or custom report) highlights documents with new approved versions that have not yet been linked to an active submission — flagging potential regulatory obligations.
Validation test: Approve a new version of a controlled document that is linked to an active submission. Verify the original Submission Document record still points to the old version. Verify the new version is flagged in the Document Tracker. This is a critical OQ test for data integrity of the submission record.
Q20Admin
How are Agency Correspondence and Regulatory Agency Commitments tracked in Vault RIM? What objects are used?
▼
Answer

Agency Correspondence in RIM is tracked as a document type within the Submission hierarchy. Incoming agency communications (Complete Response Letters, Information Requests, FDA letters) are stored as documents linked to the relevant Application/Submission.

Regulatory Commitments are tracked as a separate object type:

  • Commitment Object — Created when an agency issues a post-approval commitment. Fields: Commitment Text, Due Date, Responsible Party, Commitment Type (Post-Marketing Study, Labeling Update, Annual Report), Status (Open, In Progress, Fulfilled, Overdue).
  • Link to Application — Each commitment linked to its parent Application (and specific Activity/Submission where it originated).
  • Fulfillment tracking — When the commitment is fulfilled (a new submission made), the Commitment record is updated to link to the fulfilling Submission and status changed to Fulfilled.

Dashboard/Reports — Regulatory Operations teams run a Commitments Due Report showing all open commitments with upcoming due dates — a critical tool for proactive compliance management.

OQ test: Create a Commitment record, set Due Date to 30 days from today, verify it appears in the "Commitments Due in 30 Days" report. Mark as Fulfilled and verify it no longer appears as open. Test overdue notification fires when Due Date passes without fulfillment.
RIM
Registrations & Product Lifecycle
Approval tracking, renewals, variations, and global registration status
Q21Both
How does Vault RIM Registrations track the lifecycle of a product approval — from initial approval through renewals and post-approval variations?
▼
Answer

RIM Registrations uses the Registered Product object to track the approval status in each market:

  • Registration Record — Created when an approval is received. Fields: Approval Date, Approval Number (e.g., NDA 123456), Approved Indication, Approved Dosage Forms, Approved Labeling version, Renewal Due Date, Responsible RA Manager.
  • Registration Status — Lifecycle: Pending Approval → Approved → Renewal Due → Renewed / Expired / Withdrawn.
  • Renewal Management — Renewal Due Date field triggers scheduled notifications (e.g., 12 months before expiry → alert RA team, 6 months → escalate). Renewal submission tracked as a new Activity linked to the Registration.
  • Post-Approval Variations — Each variation (Type IA, IB, II for EMA; CBE-30, PAS for FDA) is a new Activity with its own submission and approval tracking. Upon approval, the relevant Registration fields are updated (e.g., new labeling version, new indication).
  • Approved Documents Snapshot — At the time of each approval/variation approval, the Registration record links to the specific approved document versions (labeling, SmPC, package insert). This creates an accurate regulatory state snapshot at each point in the product lifecycle.
Q22Admin
How do you configure RIM to support both FDA (US) and EMA (EU) submission types simultaneously for the same product? What are the key configuration differences?
▼
Answer

RIM handles multi-region submissions through country-specific configuration layered on the same object model. Key differences to configure:

Application Type Picklists:

  • US: NDA, BLA, ANDA, IND, NDA-Supplement (CBE-0, CBE-30, PAS), Annual Report
  • EU: MAA, Type IA Variation, Type IB Variation, Type II Variation, Line Extension, Renewal

Submission Format fields:

  • US: eCTD (NDA/BLA), paper, non-eCTD electronic (for INDs)
  • EU: eCTD via CESP portal, XEVMPD data submission for EMA product database

CTD Section mapping — Same CTD structure (M1–M5) but M1 is region-specific. US M1 (1.0 cover letter, 1.12 proposed labeling) differs from EU M1 (1.0 regional admin forms, 1.3.1 SmPC, 1.3.2 Labeling). Configure separate Submission Document Type templates per region.

Regulatory Agency records: Configure Agency object records for FDA, EMA, MHRA, Health Canada, etc., with their specific contact info, portal submission methods, and regulatory timelines.

Best practice: Use Submission Templates in RIM to pre-populate the standard set of required documents for each Application Type + Country combination. A US NDA template pre-creates the required CTD sections so RAs don't build from scratch each time.

Publishing & eDossier Management
eCTD publishing, STF, sequence management, and electronic submission
Q23Admin
Explain the eCTD publishing process in Vault RIM Publishing. What is the role of the Submission Table of Figures (STF) and how does Vault manage sequence numbering?
▼
Answer

eCTD (Electronic Common Technical Document) is the international regulatory data standard for structuring drug applications. Vault RIM Publishing manages the assembly process:

Submission Table of Figures (STF) / Submission Table of Content:

  • Each document in the submission is assigned metadata: CTD section (e.g., 3.2.P.1), document title, file name, operation (New/Replace/Delete). This metadata tells the eCTD backbone where to place each document.
  • In Vault, the STF metadata is captured on the Submission Document record and validated against the eCTD specification before publishing.

Sequence Numbering:

  • eCTD submissions use sequential sequence numbers (0000 for initial, 0001, 0002, etc. for subsequent submissions).
  • Vault tracks the sequence history per Application. When a new submission is published, Vault auto-assigns the next sequence number, preventing gaps or duplicates which would fail agency validation.

Publishing workflow:

  1. Publisher assembles the STF — links documents to CTD sections.
  2. Vault validates the eCTD package structure (XML backbone validation, file naming checks).
  3. Published eCTD is stored in RIM Submissions Archive.
  4. Publisher exports the eCTD package and submits via FDA ESG, EMA CESP, or regional portals.
  5. Agency acknowledgment receipt stored in Vault as a document linked to the Submission record.
Q24Val Lead
What are the specific OQ/PQ test cases required to validate RIM Publishing? What are the most commonly missed test scenarios?
▼
Answer

Required OQ test cases for RIM Publishing:

  • Sequence numbering — Publish sequence 0000 (initial), verify sequence 0001 is assigned for the next submission. Verify no duplicate sequence numbers are possible.
  • eCTD backbone validation — Publish a package with an intentional error (missing required section), verify Vault rejects with appropriate error message.
  • Replace operation — Document previously included in sequence 0000 is revised. Include in sequence 0001 with operation=Replace. Verify eCTD XML correctly marks the old leaf as superseded and new leaf as the current document.
  • Delete operation — Verify a document can be deleted from the eCTD and that the XML backbone correctly marks it as deleted.
  • Archive integrity — Published eCTD stored in Submissions Archive; verify file hashes/checksums to confirm the archived package matches what was published (no corruption).
  • Access control — Only Publishing-role users can initiate and complete a publish action. Verify QA-only users cannot publish.

Commonly missed scenarios:

  • Testing the partial replace scenario — where only some sections of a large submission are updated.
  • Verifying Submissions Archive is read-only post-publish — no user should be able to edit an archived eCTD.
  • Testing agency acknowledgment receipt storage and linkage to the Submission record.
  • Rendition generation — Verify PDFs are correctly generated from Word/Excel source files during publish.

Q25Val Lead
Is Vault RIM a GxP system requiring full CSV? How do you determine which RIM modules are GxP and which are not? What regulations govern it?
▼
Answer

This is a critical scoping question. Vault RIM is not uniformly GxP — you must assess each module:

  • RIM Submissions Archive — GxP (High Priority): Contains the official record of what was submitted to regulatory agencies. Regulatory records must meet data integrity requirements. 21 CFR Part 11 / EU Annex 11 applicable. Full CSV required.
  • RIM Registrations — GxP: Tracks marketing authorization status. Incorrect data here can lead to unlicensed product on market or missed renewal deadlines — regulatory compliance risk. CSV required.
  • RIM Publishing — GxP: Generates the eCTD submitted to agencies. Integrity of the published package is critical. Part 11 applicable (audit trail of who published what and when). CSV required.
  • RIM Submissions (planning) — Arguable: The planning and tracking functions are lower risk, but if the system is the system of record for submission dates and regulatory commitments, treat as GxP.

Governing regulations:

  • 21 CFR Part 11 (Electronic Records/Electronic Signatures — US)
  • EU GMP Annex 11 (Computerised Systems — EU)
  • ICH Q10 (Pharmaceutical Quality System)
  • FDA's eCTD guidance (M8 — eCTD technical specification)
  • EMA eCTD guidance and CESP submission requirements
Q26Val Lead
RIM has a dependency on the regulatory agency portals (FDA ESG, EMA CESP). How do you validate the interface between Vault RIM and these external portals?
▼
Answer

This is an interface validation challenge — you're testing a connection to an external system you don't control.

Validation approach:

  1. Scope definition: Your responsibility is to validate what Vault generates and sends. The agency portal's internal processing is outside your validation scope (but you document this boundary explicitly).
  2. Test environment consideration: FDA ESG has a Test ESG gateway for exactly this purpose. EMA CESP has a test environment. Your OQ should test the actual electronic submission to the test gateway — not just a mock.
  3. OQ test cases:
    • Submit a test eCTD package to FDA Test ESG → Receive and archive the Technical Acknowledgment (TA) and Validation Acknowledgment (VA) → Verify they are stored in the correct Submission record in Vault.
    • Test error handling: submit a malformed package → verify error is returned, logged, and the submission is not marked as "submitted" in Vault until a clean VA is received.
    • Verify digital certificate used for FDA ESG authentication is configured correctly and certificate expiry is tracked.
  4. Acknowledgment receipt workflow — Verify the workflow assigns a task to the publishing team to review and file the TA/VA received from the agency.
Certificate management: FDA ESG requires a digital certificate (obtained from FDA) for authentication. Certificate expiry must be tracked in Vault (or via another system) and renewed before expiry. A lapsed certificate prevents electronic submissions — this is a regulatory compliance risk you must include in your periodic review checklist.

Admin Best Practices & Governance
Admin controls, configuration documentation, and day-to-day governance
Q27Admin
What Admin controls should be in place to prevent unauthorized configuration changes in a validated Vault? Who should have System Admin access?
▼
Answer

Vault Admin access is extremely powerful — an admin can change lifecycles, permissions, field definitions, and object configurations with immediate effect in production. This must be tightly controlled.

Admin access controls:

  • Minimum necessary access: Only designated System Admins have Admin panel access. Business users, RA scientists, and QC analysts should NEVER have Admin access.
  • Named individuals: Admin access should be granted to 2–4 named individuals (primary + backup). No functional or shared admin accounts.
  • Separation of duties: The person making a configuration change should not be the same person who approves the change control authorizing it.
  • Admin audit trail review: Periodically review Admin Activity Audit Log (Admin → Audit Trail → Admin Actions) to confirm all admin-level changes have corresponding approved change controls. This is a key periodic review deliverable.
  • DEV-only admin rule: Wherever possible, config changes are made ONLY in DEV by the admin, then promoted via migration package. Direct PROD admin changes are emergency only and require retrospective change control documentation.
Common audit finding: Admin accounts found for former employees, or business users found with admin access "because they needed it once." Quarterly admin access review is a best practice that prevents this.
Q28Admin
How do you use Vault's built-in tools — Vault Loader, API, and the Vault Java SDK — and in what admin scenarios would you use each?
▼
Answer

Vault provides multiple integration and bulk-operation tools, each suited to different scenarios:

  • Vault Loader (CSV-based) — GUI or CLI tool for bulk create/update/delete of documents and objects via CSV files. Best for: Data migrations, bulk user updates, mass document metadata updates. Requires admin knowledge of object field API names. Produces a results CSV showing success/failure per row. Use when: one-time or infrequent bulk data operations.
  • Vault REST API — Full programmatic access to all Vault operations: CRUD on objects, document upload, workflow management, user management. Requires authentication (API credentials). Best for: System integrations (LIMS → Vault, ERP → Vault), automated data sync, building custom integration middleware. Use when: ongoing automated data exchange with other systems.
  • Vault Java SDK (Vault Extensions) — For building custom server-side logic in Vault: custom triggers (execute code on object save), custom actions (button-triggered logic), custom jobs (scheduled automation). Runs inside Vault's sandbox environment. Use when: business logic too complex for native configuration, custom validation rules, complex auto-calculations.
Validation implication: Any use of Vault Loader for GxP data migration must be validated (the migration itself is a validated activity). Vault Extensions (custom code) are Category 5 (custom software) and require their own code review, unit testing, and validation testing. Never deploy custom Vault SDK code to production without validated testing in QA vault first.
QMS + RIM
Electronic Signatures & 21 CFR Part 11
E-sig configuration, Part 11 compliance, and what inspectors examine
Q29Both
How does Vault implement 21 CFR Part 11 compliant electronic signatures? What are the technical requirements and how do you configure and test them?
▼
Answer

21 CFR Part 11 requires electronic signatures to be: unique to one individual, not reused or reassigned, linked to their respective electronic record, and include the printed name, date/time, and meaning of signature.

Vault E-Signature configuration:

  • Signature meaning — Configured per workflow task or lifecycle transition. E.g., QA Approval signature meaning: "I approve this document as compliant with applicable procedures." This is displayed to the signer before they sign and permanently recorded in the audit trail.
  • Authentication — Vault requires the user to re-enter their password at the point of signing (even though they're already logged in). This is the 21 CFR Part 11 §11.200 "biometric/non-biometric" requirement — re-authentication at time of signing.
  • Signature record — The signed record stores: Username (unique ID), Full printed name, Date/time (UTC, then converted to user's timezone for display), Signature meaning, Document version that was signed. This is all part of the audit trail and is tamper-proof.
  • Signature manifest — Documents with e-signatures generate a printable Signature Page (signature manifest) showing all signature events. This is included when generating a certified copy of the record.

OQ test cases:

  • Attempt to sign with incorrect password → system rejects. Correct password → signature recorded.
  • Verify signature record shows correct username, printed name, date/time, and meaning.
  • Verify the audit trail entry for the signature cannot be edited or deleted.
  • Sign as User A, log out, log in as User B, verify User B cannot sign with User A's credentials.
  • Verify printed signature manifest matches audit trail.
Q30Val Lead
What does a 21 CFR Part 11 gap assessment for a new Vault implementation look like? What are the most common gaps found?
▼
Answer

A Part 11 gap assessment maps each §11.10 (controls for closed systems) and §11.200 (electronic signature) requirement against your Vault configuration. Format is a compliance matrix:

§11.10(a) — Validation: COMPLIANT — System validated per CSV protocol. §11.10(b) — Accurate/complete copies: COMPLIANT — Vault generates printable records with e-sig manifest. §11.10(c) — Record protection/retrieval: COMPLIANT — Vault backup/DR, retention configured. §11.10(d) — System access limited to authorized users: COMPLIANT — Permission sets, unique IDs, password policy. §11.10(e) — Audit trail: COMPLIANT — System-generated, tamper-proof, reviewed per schedule. §11.10(f) — Operational system checks: REVIEW — Check all mandatory fields enforced. §11.10(g) — Authority checks: COMPLIANT — Role-based access prevents unauthorized signing. §11.10(h) — Device checks: COMPLIANT — Vault session management configured. §11.10(i) — Training: GAP — Training records not yet in the validated LMS. [ACTION] §11.10(j) — Policies: GAP — Part 11 policy SOP not yet drafted. [ACTION] §11.10(k) — Controls for distribution/use: COMPLIANT — Controlled document module active.

Most common gaps found:

  • No formal Part 11 Policy SOP signed by management.
  • Users have not signed a Part 11 Commitment Statement (acknowledging their e-sig is legally binding).
  • Audit trail review cadence not defined — no SOP, no evidence of periodic review.
  • Shared accounts exist (even for admins — "we share the admin login for convenience").
  • Password policy configured but not formally documented as meeting §11.300 requirements.
QMS + RIM
Vendor Upgrades, Releases & Long-Term Maintenance
Managing Veeva's release cycle, long-term validation burden, and strategic maintenance
Q31Both
Veeva releases major updates ~3 times per year. How do you build a sustainable, resource-efficient release management program that doesn't burn out the validation team?
▼
Answer

This is a strategic operations question as much as a technical one. Key pillars:

  1. Tiered regression library — Maintain 3 levels: Tier 1 (5–10 smoke tests, run in 30 minutes, every release), Tier 2 (20–30 targeted tests, run when release notes indicate impact), Tier 3 (full regression, run for major releases or when multiple GxP functions impacted).
  2. Impact assessment template — Standardized spreadsheet mapping each Veeva release note line item against your configured GxP functions. Built once, reused every release. Admin reviews release notes and color-codes (Green/Amber/Red) impact level in ~2 hours per release.
  3. Test automation — Invest in Vault's built-in test utilities or third-party tools (Selenium, or Veeva's own Vault Direct Data API) to automate Tier 1 smoke tests. Initial investment saves hours per release cycle.
  4. Business user partnership — Train power users in each department (QA, RA, QC) to run their own functional smoke tests after each release. Formalize this as a UAT step with a simple pass/fail checklist. Reduces load on the validation team.
  5. Veeva sandbox preview window — Veeva provides sandbox access 4–6 weeks before production release. Use this window proactively — don't wait until the week before production deployment.
  6. Release calendar integration — Subscribe to Veeva's release schedule and block time in the validation team's calendar each release cycle. Treat it as a recurring operational activity, not a surprise event.
Metric to track: Time from sandbox availability to completed regression sign-off. Target: under 3 weeks. If consistently over, your regression library needs streamlining or you need additional resources.
Q32Val Lead
An FDA inspector arrives and asks: "Show me evidence that your Vault QMS has been maintained in a validated state since go-live three years ago." Walk me through what you present.
▼
Answer

This is the ultimate "prove it" question. Your response is a structured walkthrough of your Validation Lifecycle Binder:

  1. "We start with our Validation Summary Report from initial go-live." — Show the signed VSR confirming IQ/OQ/PQ complete, RTM 100% coverage, all deviations closed. This establishes the validated baseline.
  2. "Here is our Change Control Register — every change to the system over 3 years." — Show the CC log: date, change description, impact assessment, approval, test evidence reference, closure date. Demonstrate no changes were made outside of change control (admin audit trail confirms this).
  3. "For each major change, here is the re-validation evidence." — Pull 2–3 significant changes and show the full package: IA → updated RTM → OQ test scripts → executed results → closure.
  4. "We also manage Veeva's quarterly releases." — Show 3 years of Release Impact Assessments and regression test records. Demonstrate that every Veeva production release was assessed and tested before reaching the live system.
  5. "Annual Periodic Reviews confirm the system remains validated." — Show 3 years of Periodic Review reports. Each signed by Business Owner + QA. Each covers: change history, incident review, access review, SOP review.
  6. "User access records." — Show current user list with roles. Show the annual access review records. Show that terminated employee accounts were disabled (pull one example).
  7. "Training records." — Show training completion report for all current users against all current SOPs. No gaps.
If you can't show this chain of evidence: Gaps are major or critical findings. The inspector will conclude the system has not been maintained in a validated state regardless of what it was at go-live. Validation is a continuous commitment, not a one-time event.
