Step 01 · Onboarding
System Categorization
& GxP Assessment
Before any validation begins, determine whether the system is GxP-relevant, classify it under GAMP 5, and assess regulatory impact. This defines your entire validation strategy.
GxP Impact Assessment

Ask these questions for any new system. If any answer is YES, the system requires formal validation.

Q1
Does the system create, modify, or archive regulated records?
Lab results, batch records, QC data, electronic submissions, audit trails — 21 CFR Part 11 / Annex 11 applicability.
Q2
Is it used to make quality or safety decisions?
LIMS releasing a batch, ERP triggering release, QMS routing deviations and CAPAs.
Q3
Does it control or monitor GMP equipment or processes?
DCS, SCADA, environmental monitoring systems, autoclave controllers.
Q4
Is it used in regulatory submissions?
Veeva Vault RIM, document management systems, CTD authoring tools.

The Validation Master Plan (VMP) or System Validation Plan (SVP) is the governing document. It defines scope, approach, responsibilities, and acceptance criteria before any testing begins.
Validation Plan — Key Components
1

System name, version, hosting environment, intended use, regulated functions, interfaces with other systems.
2

3

Risk-based approach statement, GAMP category, IQ/OQ/PQ breakdown, vendor documentation leveraged.
4
Roles & Responsibilities
Validation owner, SMEs, IT, QA reviewer/approver, vendor contacts. RACI matrix often included.
5
Deliverables List
All documents to be produced: URS, FRS, DS, RTM, IQ, OQ, PQ protocols and reports, SOPs, training records.
6
Risk Assessment
System-level risk assessment, GxP risk score, identification of critical functions requiring higher test rigor.
7

Projected start/end dates, phase gates, QA review checkpoints, go-live date.

Step 03 · Validation Phase
IQ — Installation
Qualification
Verifies that the system is installed correctly and according to vendor specifications, and that the environment meets all prerequisites before functional testing begins.
IQ Protocol Contents
🖥️
Hardware Verification
Server specs vs. vendor minimum requirements. CPU, RAM, disk, network configuration. Rack location, physical access controls.
⚙️
Software Installation
Application version confirmed, installation path documented, license keys recorded, all components installed per runbook.
🔧
Configuration Baseline
Initial configuration screenshots, config files archived, database schema version, service account setup, firewall rules documented.
🌐
Network & Interfaces
Network diagrams, interface connections to other systems, SSL/TLS certificates, port configurations, data flow diagrams.
📁
Backup & DR
Backup schedule configured and tested, disaster recovery procedures documented, RTO/RPO confirmed for the system.
🛡️
Security Configuration
Audit trail enabled and tested, e-signature configuration, password policy settings, inactivity timeout confirmed.
IQ Report — Acceptance Criteria
PASSED: All IQ test steps executed. All actual results match expected. No open critical/major deviations. Deviations (if any) have approved resolutions. IQ Summary Report signed by Validation Owner and QA.

Step 04 · Validation Phase
OQ — Operational
Qualification
Verifies that the system functions as designed across its full operational range — all configured features, business logic, and system behaviors are tested against the FRS and design specification.
OQ Scope — What Gets Tested
F1
Core Functional Workflows
Every configured business workflow per the FRS. For a LIMS: sample login, analysis assignment, result entry, result review, approval, reporting. For Veeva: document lifecycle (Draft → Review → Approved → Superseded).
F2
User Roles & Permissions
Verify each role can only perform permitted actions. Negative testing critical: an Analyst cannot approve results, a Read-Only user cannot create records.
F3
Electronic Signatures & Audit Trail
E-sig triggers correct (meaning, date/time, user ID). Audit trail captures all creates/edits/deletes. Audit trail is tamper-evident and not modifiable by end users.
F4
Calculations & Business Rules
Configured calculations verified with known inputs/outputs. OOS/OOT flags trigger at correct thresholds. Workflow routing logic verified.
F5
Interface / Integration Testing
Data transfers between integrated systems verified. LIMS ↔ ERP, Veeva ↔ regulatory portals. Verify data completeness, integrity, and error handling.
F6
Negative / Error Path Testing
System behavior on invalid inputs, mandatory field enforcement, duplicate record handling, session timeout, and concurrent user conflicts.
OQ Test Script Structure
Each test script must contain: Test Case ID · Requirement ID (links to RTM) · Objective · Prerequisites / Test Data · Step-by-step actions · Expected Result (pre-defined) · Actual Result (filled during execution) · Pass/Fail · Tester + Date · Reviewer sign-off.
Step 05 · Validation Phase
PQ — Performance
Qualification
Demonstrates that the system consistently performs as intended in the actual operational environment, using real or representative data, by actual end users under real-world conditions.
PQ vs. OQ — Key Difference
⚗️ OQ Focus
Configured features tested by the validation team. Synthetic test data. Controlled environment. Proves the system can do what it's configured to do.
🏭 PQ Focus
End-to-end business processes run by actual users. Production-like or real data. Proves the system will work in day-to-day operation.
PQ Execution Elements
P1
End-to-End Process Scenarios
Complete business scenarios from initiation to final output. E.g., Full sample lifecycle in LIMS from receipt to COA generation. Full document lifecycle in Veeva from draft to regulatory submission.
P2
Volume / Load Testing (if applicable)
If system processes high data volumes, verify performance under expected peak load. Response times documented and compared to acceptance criteria.
P3
Executed by End Users
Business users (not the validation team) run the tests. This also serves as hands-on training. Observed by the validation lead.
P4
Data Integrity Verification
Verify that data entered is accurately stored, retrieved, reported, and archived. ALCOA+ principles confirmed: Attributable, Legible, Contemporaneous, Original, Accurate.
ALCOA+ Principles
A — Attributable
Who performed the action and when. Unique user IDs, no shared logins, audit trail captures username.
L — Legible
Data is readable and permanent. No overwriting without audit trail. Timestamps in unambiguous format.
C — Contemporaneous
Recorded at the time the activity occurs, not retrospectively.
O — Original
First capture is preserved. Source data retained. No transcription errors from paper to electronic.
A — Accurate
Data reflects the actual event. Calculations verified. No rounding errors. Instrument calibration confirmed.
+ Complete, Consistent, Enduring, Available
All data present, time-consistent, retained for required retention period, accessible to authorized users.
Step 06 · Validation Phase
UAT & Requirements
Traceability Matrix
User Acceptance Testing is the final gate before go-live. The RTM ties every requirement to its test evidence. These two together form the core of your validation package.
UAT Process
1
UAT Planning
Business users define acceptance criteria. UAT scripts written based on real workflows. Test environment mirrors production. UAT plan approved by Business Owner and QA.
2
UAT Execution
Business stakeholders execute scenarios. Issues logged in defect tracker with priority (Critical / Major / Minor). No critical or major open defects at go-live.
3
Defect Resolution Cycle
Developer fixes → re-test in same environment → regression testing for impacted areas → QA review of fix. If fix requires configuration change, assess whether re-validation is required.
4
UAT Sign-Off
Business Owner signs UAT completion. All critical/major defects resolved. Minor defects either resolved or accepted with documented justification.

Change Impact Assessment — Key Questions
IA1
What is the nature of the change?
Configuration change? Software upgrade? New interface? SOP update? Infrastructure change? Hardware replacement?
IA2
What GxP functions are affected?
List all impacted validated functions. Assess risk level: High / Medium / Low.
IA3
Does the change require re-validation?
High-risk changes to validated functions typically require OQ/PQ re-execution. Low-risk changes may need only regression testing or impact assessment documentation.
IA4
Does the change require SOP updates?
If business workflows change, affected SOPs must be updated, reviewed, approved, and users retrained before go-live.
IA5
Is rollback possible?
Define rollback plan before implementation. Backup taken and verified. Rollback tested in non-prod if high risk.

Step 08 · Enhancement Requests
User Enhancement
Request Process
Enhancement requests (ERs) are formal requests for new features or functional improvements from end users. Unlike bug fixes, ERs represent planned changes that enhance system capabilities and must follow a structured lifecycle.
ER Intake & Prioritization
1
Enhancement Request Submission
User submits via formal channel (QMS, ITSM like ServiceNow, or change management module). ER must include: business justification, current state vs. desired state, regulatory impact if any, priority (High/Med/Low), and requesting department.
2
Business Owner Review
Business Owner confirms business need, checks for duplicates or related open changes, assigns initial priority.
3
Technical Feasibility Assessment
System/IT Owner assesses: Is this configurable or does it require custom development? Estimated effort. Dependencies. Impact on existing validated functions.
4
GxP / Validation Impact Assessment
Validation team or QA determines: Does this affect validated functions? What re-testing is required? RTM update needed? SOP/training update required?
5
Change Control Board (CCB) Approval
CCB reviews ER with impact assessment. Approves for implementation with defined scope, timeline, and re-validation requirements. Formally opens a Change Control record.

Environment Strategy: GxP systems must have separate DEV / TEST / UAT / PROD environments. Changes must never be made directly in PROD without following the promotion workflow. Environment qualification (EQ) should be documented for each environment.
Vendor-Managed Systems (e.g., Veeva Vault)
SaaS / Cloud considerations: For vendor-managed systems (Veeva, Salesforce, cloud LIMS), vendor patches and upgrades are pushed on a schedule. You must: (1) monitor vendor release notes, (2) assess GxP impact of each release, (3) run regression testing in sandbox before accepting in production, (4) maintain documentation of your assessment and testing. This is critical — vendor upgrades ARE changes to your validated system.
Step 09 · Ongoing Operations
Periodic Review &
Revalidation
Validated GxP systems require ongoing oversight. Periodic review confirms the system remains in a validated state, identifies configuration drift, and captures the cumulative impact of changes since last review.
Periodic Review Components
PR1
Review Frequency
Typically annual, though risk-based frequency is acceptable. High-criticality systems (e.g., batch release LIMS) reviewed annually. Lower-risk systems may be 18–24 months with justification.
PR2
Change History Review
Review all changes implemented since last periodic review. Confirm all changes went through change control. No undocumented changes (configuration drift check).
PR3
Incident / Deviation Review
Review all system-related incidents, deviations, and CAPAs. Trends identified. Open CAPAs assessed for resolution status.
PR4
User Access Review
Review all active user accounts. Confirm roles are appropriate. Remove or disable stale accounts (terminated employees, role changes). Review privileged access.
PR5
Validation Status Confirmation
Declare the system remains in a validated state (or identify what re-validation is needed). Sign-off by System Owner, Business Owner, and QA.
PR6
SOP Effectiveness Review
Are SOPs being followed? Are there recurring issues that suggest SOP gaps? Confirm SOP revision dates are current.
Revalidation Triggers
⚠️ Always Requires Re-validation
  • Major software version upgrade
  • Migration to new hardware or cloud platform
  • Major new functionality for GxP use
  • Change to audit trail mechanism
  • Change to e-signature implementation
📋 Risk-Based Assessment
  • Minor version / patch upgrades
  • Configuration changes in GxP areas
  • OS or database upgrades
  • New interface to another system
  • Significant SOP changes

SOPs, Training
& Competency
SOPs and training records are not administrative overhead — they are part of the validated state. An auditor will check that users are trained, SOPs are current, and training is documented before looking at anything else.
Required SOPs for a Validated System
👤
Access Control SOP
User provisioning, role assignment, access removal (leavers/movers), periodic access review, emergency access procedure.
🔄
Change Control SOP
How changes are requested, assessed, approved, implemented, and documented. Environment promotion workflow.
💾
Backup & Recovery SOP
Backup schedule, verification, restoration testing frequency, off-site storage, recovery time objective.
🚨
Incident Management SOP
How system incidents are logged, triaged, investigated, and resolved. Link to deviation management system.
📊
Periodic Review SOP
Frequency, scope, responsibilities, and documentation requirements for periodic review of validated systems.
📋
System Use SOPs
Work instructions for each GxP workflow: how to enter data, review, approve, generate reports, handle errors.
Training Program Requirements
T1
Role-Based Training Curriculum
Each role has a defined training curriculum. Analysts train on different SOPs than Reviewers or Admins. Training matrix maintained.
T2
Training Before System Access
Users must complete required training and demonstrate competency before being granted access. Training records must be complete before access request is approved.
T3
Retraining on Change
When SOPs are updated due to system changes, affected users must be retrained. Go-live of a change cannot precede completion of retraining.
T4
Training Record Retention
Training records retained for the life of the system plus applicable regulatory retention period. Typically electronic, stored in LMS or QMS with audit trail.

Step 12 · End of Life
System Retirement &
Decommissioning
Decommissioning a GxP system is a formal, validated activity. Data must be migrated or archived in a retrievable format for the required regulatory retention period before the system is shut down.
Decommissioning Plan — Key Elements
D1
Retirement Decision & Justification
Formal decision with business rationale. New system to replace (if applicable) must be fully validated before old system is decommissioned.
D2
Data Retention Assessment
Identify all regulated records in the system. Determine regulatory retention requirements (FDA: batch records 3 yrs post expiry, for example). Define archive format and access method for the retention period.
D3
Data Migration or Archive
Migrate to new system OR archive to validated archive solution. Data migration validation required if migrating to a new GxP system (verify completeness, integrity, no corruption).
D4
Validation Package Archive
All validation documentation (IQ/OQ/PQ, SOPs, change records, training, periodic reviews) must be archived and retrievable for the retention period, even after the system is gone.
D5
Decommissioning Report
Formal report confirming: data migrated/archived successfully, system shut down, licenses returned, infrastructure decommissioned, documentation archived. Signed by System Owner, Business Owner, and QA.
Key Principle: You must be able to reproduce a GxP record from a decommissioned system during a regulatory inspection 10+ years later. Plan your archival strategy accordingly — including the ability to render archived data in human-readable form even when the original software no longer exists.

🔴 Event Detected — Duplicate WeightIDs in ELN LabX

During routine QC sample processing, a lab analyst noticed that two separate weighing tasks had been created with the same WeightID in the ELN LabX balance module. This meant two distinct physical weighing events shared a single identifier — a direct violation of ALCOA's Attributable and Original principles. The issue was escalated to the LIS Lead and QA the same day.

🟠 Immediate Containment

As soon as the issue was confirmed, new weighing task creation in the affected LabX workflow was restricted to prevent additional duplicates from being generated. All records in the system were queried against the WeightID field to identify the full scope of duplicates. A list of affected records was compiled and flagged for data review. No data was deleted or altered at this stage — containment only.

🟣 Deviation Record Opened

A formal unplanned deviation was opened in the QMS the same day. Classification: Unplanned, GxP-critical, High severity. The record captured: system name (ELN LabX), date of discovery, who identified it, description of the event, initial scope estimate, and immediate containment actions taken. The deviation was assigned to the LIS Lead as investigation owner, with QA as approver.

🔵 Root Cause Investigation — 5-Why Analysis
  • Why 1: Duplicate WeightIDs existed → Users could manually enter any WeightID value with no validation.
  • Why 2: No validation on entry → WeightID field had no uniqueness constraint enforced at the application or database layer.
  • Why 3: No constraint configured → The field was designed to be unique by convention; enforcement was never built into the configuration.
  • Why 4: Not caught in validation → The original OQ test suite did not include a negative test for duplicate WeightID entry.
  • Root cause: Configuration gap — missing uniqueness constraint — combined with an incomplete OQ test library that lacked this negative scenario.
🟢 Impact Assessment

A systematic query of the LabX database identified all WeightID duplicates across the affected date range. Each duplicate pair was cross-referenced against original paper weight logs to determine which record was correct and which was erroneous. Data integrity risk was assessed as High — if results had been reported from an incorrectly identified weighing event, it could affect analytical calculations and batch release decisions. No batch releases were found to have been directly impacted, but the potential risk was documented.

🟢 QA Review & CAPA Threshold

QA reviewed the deviation, root cause, and impact assessment. Given the data integrity risk classification (ALCOA violation, potential impact on analytical results) and the systemic nature of the root cause (configuration gap, not a one-off human error), the risk score met the organization's CAPA threshold. QA approved the deviation investigation and triggered the CAPA auto-generation, linking it directly to this deviation record.

🔴 CAPA Record — Structure

The linked CAPA record was structured with two distinct workstreams:

Corrective Action

Fix the immediate problem — add a uniqueness constraint to the WeightID field so the system enforces uniqueness at the point of entry. Implemented via formal change control.

Preventive Action

Prevent recurrence systemically — update the config review SOP to require a uniqueness check on all identifier fields for any new workflow. Add a negative test OQ case to the standard regression library.

🟢 Corrective Action — Config Change Control

A Change Control was raised to add a database-level uniqueness constraint on the WeightID field in the ELN LabX configuration. The change went through the standard promotion workflow: configured in DEV → OQ regression tested in QA vault (negative test: attempt to save a duplicate WeightID — verified system rejects with a clear validation error) → promoted to PROD after QA approval. Post-deployment verification confirmed the constraint was active. Affected duplicate records were remediated under a documented data correction process — each correction logged in the audit trail with the original value, new value, reason, and approver.

🟣 Preventive Action — SOP & OQ Test Library Update

Two preventive actions were implemented: (1) The system configuration review SOP was updated to include an explicit step requiring review of all identifier fields for uniqueness constraint enforcement during any new workflow buildout or enhancement activity. (2) A new OQ test case was added to the standard regression library: "Attempt to create a duplicate [identifier field] — verify system rejects with appropriate error message." This test now runs on every future release regression cycle for LabX and was added as a template requirement for other ELN configurations.

🔵 Effectiveness Verification

After PROD deployment, the OQ effectiveness test was re-executed: attempt to create a new weighing task with an already-existing WeightID. The system rejected the entry with a clear validation error message. The test execution record (tester, date, actual result, pass status) was attached to the CAPA as effectiveness evidence. A 30-day monitoring period confirmed no new duplicate WeightID records appeared in the system after the fix was deployed.

🟠 Change Control Closure

The Change Control record was closed after: (1) configuration deployed to PROD, (2) post-deployment verification step confirmed and documented, (3) affected records remediated, (4) OQ evidence attached. The CC record was linked back to the CAPA record, so both records carry cross-references to each other. QA reviewed and signed off the CC closure. The CAPA's corrective action task was marked complete, referencing the CC record number.

✅ Deviation & CAPA Closed

Both records were formally closed with QA electronic signature. Closure criteria confirmed: root cause identified and documented; corrective action implemented and verified effective; preventive action implemented (SOP updated, OQ test added); all affected records remediated; change control closed; 30-day monitoring complete with no recurrence. All records retained in the QMS per the applicable data retention policy. At the next periodic review, QA signed off that actions were sustained.

Regulatory principle: A deviation is only closed when you can prove the root cause has been fixed AND it won't happen again. Evidence of effectiveness — not just completion of actions — is what closes a CAPA.
Key Principles — Deviations & CAPAs
P1
Same-day reporting
Deviations must be opened the same day the event is identified. Late reporting is itself a GxP finding. If you're unsure whether something is a deviation, open it — it's easier to close an unnecessary record than to justify late reporting.
P2
Containment before investigation
Stop the bleeding first. Restrict access, flag affected records, pause the affected process if needed. Document containment actions in the deviation record before moving to root cause analysis.
P3
Root cause ≠ the immediate cause
The root cause is the deepest systemic reason the event was possible. "User error" is almost never the root cause — it's a symptom. Dig into why the system or process allowed the error to occur.
P4
CAPA threshold is defined in your SOP
Not every deviation requires a CAPA. Your QMS SOP defines the risk scoring criteria (severity × likelihood × detectability) that triggers a CAPA. High-risk GxP data integrity events almost always meet the threshold.
P5
Effectiveness verification is mandatory
You cannot close a CAPA based on completing the actions alone. You must demonstrate the actions were effective — the deviation scenario does not recur, and the root cause has been eliminated. Test it, document it, attach it as evidence.
P6
Cross-reference everything
Deviation → CAPA → Change Control must all carry each other's record numbers. An auditor tracing a finding must be able to follow the chain from detection all the way through to verified closure without gaps.

Deviation Lifecycle

Deviation Types — Know Before You Open
⚡ Unplanned Deviation

Something happened that was not anticipated or approved in advance. Equipment failure during a run, an analyst following a different procedure, a system producing unexpected data. Must be reported the same day discovered. No pre-approval exists — the deviation IS the notification.

📋 Planned Deviation

A known, intentional departure from a procedure — requested and approved in advance. Example: testing a new reagent lot under slightly different conditions than the SOP specifies, with QA pre-approval. The deviation record is opened BEFORE the activity occurs. Risk assessed and approved prior to execution.

Key rule: If you're unsure whether something is a deviation, treat it as one and open the record. It is far easier to close an unnecessary deviation record with a documented justification than to defend why a potential GxP event was not reported. "When in doubt, document."

🔴 Event / Non-Conformance Occurs

A deviation starts the moment something does not go as the approved procedure, specification, or process requires. This could be an instrument failing mid-run, an analyst making an error, a system producing an unexpected output, or a sample being processed under out-of-spec conditions. For planned deviations, the record is opened before the activity occurs, with pre-approval from QA. For unplanned, the clock starts from the moment of discovery — not from when the event actually happened if that was earlier.

⚫ Non-GxP Event — Log Only

If the event occurs in a non-GxP process (e.g., an administrative process with no impact on regulated data, equipment, or products), a formal deviation record may not be required. However, good practice still requires documenting the event in a log or note-to-file, capturing what happened and why it is classified as non-GxP. This documentation protects you if the classification is later questioned — you have a record of the decision and the reasoning.

Caution: Be conservative. If there is any doubt about whether the process is GxP-relevant, open the deviation. An unnecessary deviation record costs time; a missing one can cost an inspection outcome.
🟣 Open the Deviation Record — Same Day

The deviation must be opened in your QMS on the same day the event is identified. Mandatory fields to complete at opening:

  • Date of event — when the deviation actually occurred (may differ from date of discovery)
  • Date of discovery — when it was identified
  • System / product / process affected
  • Description — factual, objective statement of what happened. No conclusions yet.
  • Deviation type — Planned or Unplanned
  • Initial severity estimate — Critical / Major / Minor
  • Initiator — person opening the record
  • Immediate containment actions taken
Common mistake: Writing conclusions or root cause in the opening description. At this stage you only know what happened — not why. Write facts only. Root cause is documented in the investigation section once it has been determined.
🟠 Immediate Containment Actions

Before investigation begins, stop the impact from spreading. Containment actions are documented in the deviation record at the time they are taken. Common containment actions:

  • Quarantine affected materials, samples, or products — label clearly "HOLD — Deviation Under Investigation"
  • Suspend the affected process or workflow until root cause is known
  • Restrict system access or data entry in the affected area
  • Preserve evidence — do not repair, clean, or restart equipment involved until documented
  • Notify relevant stakeholders (supervisor, QA, Regulatory Affairs if batch release is affected)
  • Flag affected electronic records — do not alter or delete any data at this stage
ALCOA principle: All data related to the deviation — including the original erroneous data — must be preserved. You may correct records through a formal data correction process with audit trail entries, but never delete original values. The original is the evidence.

🔵 Root Cause Investigation

The investigation must identify the true root cause — the deepest systemic reason the deviation was possible. Two primary tools:

5-Why Analysis

Start with the observable event and ask "why" repeatedly until you reach the system- or process-level cause. Stop when the answer is something that can actually be fixed. Example: WeightID duplicate → no validation → no constraint configured → not in OQ test library → SOP gap. Root cause: SOP did not require uniqueness constraint checks.

Ishikawa (Fishbone)

Useful for complex events with multiple potential causes. Categories: Man, Machine, Method, Material, Measurement, Environment (6M). Map all contributing factors before identifying the primary root cause. Better for manufacturing deviations than software/system ones.

Key output: The investigation report must state: (1) confirmed root cause, (2) contributing factors, (3) evidence reviewed, (4) what was ruled out and why. "Unknown" is never an acceptable root cause — keep investigating until you have a clear, actionable answer.
🟢 Impact Assessment

The impact assessment answers: how far did this deviation reach, and what are the consequences? Key questions:

  • Records affected: Which specific GxP records, batches, samples, or results are impacted?
  • Time window: How long has the root cause condition existed? Are older records potentially affected?
  • Patient / product safety: Is there any risk to product quality, patient safety, or regulatory compliance?
  • Batch release impact: Are any batches currently pending release that rely on affected data?
  • Regulatory notification: Does the impact require notification to a regulatory authority?
  • Remediation needed: Do affected records require data correction under a formal correction process?
🟡 CAPA Threshold Check

Your QMS SOP defines a risk score threshold above which a CAPA is mandatory. The threshold is typically based on a combination of severity, likelihood of recurrence, and detectability. A CAPA is almost always required when:

  • The root cause is systemic (configuration gap, SOP gap, training gap) — not an isolated one-time human error
  • The same root cause has appeared in previous deviations (trend)
  • Data integrity was compromised or potentially compromised
  • The deviation is classified Critical or Major
  • Regulatory action risk exists
When CAPA is NOT required: A one-off, isolated minor deviation with a clear human error root cause, no systemic cause, and no data impact may be closed without a CAPA — with a documented justification for why preventive action beyond retraining is not warranted. This justification must be QA-approved.
🟢 QA Review

QA independently reviews the full deviation package before approving the investigation findings:

  • Is the root cause well-evidenced and credible? Is further investigation needed?
  • Is the impact assessment complete? Are all affected records identified?
  • Are the proposed corrective actions appropriate to the root cause?
  • Is the severity classification correct?
  • Has the CAPA threshold been correctly applied (triggered or not triggered with justification)?
  • Are due dates for corrective actions realistic and compliant with SOP timelines?

If QA rejects the investigation: the deviation returns to the investigator for additional root cause work. The rejection reason and requested additional investigation are documented in the QMS record. This loop is visible in the audit trail — rejected investigations are not unusual, they demonstrate the QA oversight process is working.

🟢 Corrective Actions

Corrective actions fix the immediate problem caused by the deviation. They are distinct from preventive actions (which are part of the CAPA). For a deviation without a CAPA, corrective actions may include:

  • Data remediation: Correct affected records through a formal data correction process. Each correction logged in the audit trail: original value, corrected value, reason, corrector, approver.
  • Equipment repair / recalibration: If equipment was involved, service and recalibrate before returning to GxP use.
  • Batch disposition: If a batch is affected, QA makes a formal disposition decision (release, quarantine, reject) with documented justification.
  • Immediate retraining: If the root cause is a knowledge gap, affected personnel are retrained on the specific procedure before resuming the activity.
🟠 Change Control — When Required

If the corrective or preventive action requires a change to a validated system configuration, an SOP, or a controlled process, a formal Change Control must be raised. The deviation and CC records are cross-linked. The deviation cannot close until the CC is completed (or the CC is formally deferred with a documented risk acceptance). Common change-requiring corrections:

  • Adding a validation rule or constraint to a system (e.g., uniqueness constraint on a field)
  • Modifying a workflow configuration in a validated LIMS or QMS
  • Revising an SOP or work instruction
  • Changing an instrument calibration procedure or frequency
🟣 SOP / Training Update

If the investigation identified an SOP gap or a training gap as a contributing factor, the relevant SOP must be revised and all affected users retrained before the deviation can close. The sequence matters:

  1. SOP revised and approved through the document control workflow
  2. Training assignments auto-generated for all users in affected roles
  3. Training completed and records filed (e-signature on training assignment)
  4. Training completion report attached as evidence to deviation
Sequence rule: Users cannot return to the activity governed by the updated SOP until training is complete. Going live with an updated SOP before retraining is itself a deviation.
🔵 Effectiveness Check

After corrective actions are implemented, you must verify they worked. The effectiveness check is not just a sign-off that the action was completed — it is evidence that the root cause has been eliminated and the deviation scenario does not recur.

  • For system changes: Re-execute the test case that replicates the deviation scenario. Document the result (system now prevents the error). Attach test execution record.
  • For process changes: Monitor the process for an agreed period (e.g., 30 days, 10 runs, next 3 batches). No recurrence of the same deviation pattern = effective.
  • For training gaps: Assessment or knowledge check after retraining. Or monitored observation of the affected activity being performed correctly.
If not effective: The deviation returns to the investigation stage. Root cause may have been incorrectly identified, or the corrective action did not address it fully. This is documented in the deviation record and the loop restarts. Repeat deviations with the same root cause are a red flag in regulatory inspections.
✅ Deviation Closed — Final Checklist

Before QA signs off closure, every item on this checklist must be complete:

  • Root cause documented, evidenced, and QA-accepted
  • Impact assessment complete — all affected records identified and remediated
  • All corrective actions completed on time (or overdue with documented justification)
  • Change Control closed (if a system or process change was required)
  • SOP updated and all affected users retrained (if required)
  • CAPA opened and linked (if risk score met threshold)
  • Effectiveness check completed with supporting evidence attached
  • No open blocking items

QA applies their electronic signature with meaning "I confirm this deviation has been fully investigated, all actions completed, and the record is approved for closure." The record is locked in the QMS — no further edits possible. Retained per the applicable data retention policy.

Retention: Deviation records must be retained for the life of the batch/product plus the regulatory retention period — typically a minimum of 3 years after batch expiry in the US. For systems, retain for the life of the validated system plus applicable period. Records must remain accessible for regulatory inspection throughout that window.
