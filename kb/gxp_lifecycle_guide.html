Step 01 · Onboarding
System Categorization
& GxP Assessment
Before any validation begins, determine whether the system is GxP-relevant, classify it under GAMP 5, and assess regulatory impact. This defines your entire validation strategy.
GxP Impact Assessment

Ask these questions for any new system. If any answer is YES, the system requires formal validation.

Q1
Does the system create, modify, or archive regulated records?
Lab results, batch records, QC data, electronic submissions, audit trails — 21 CFR Part 11 / Annex 11 applicability.
Q2
Is it used to make quality or safety decisions?
LIMS releasing a batch, ERP triggering release, QMS routing deviations and CAPAs.
Q3
Does it control or monitor GMP equipment or processes?
DCS, SCADA, environmental monitoring systems, autoclave controllers.
Q4
Is it used in regulatory submissions?
Veeva Vault RIM, document management systems, CTD authoring tools.
GAMP 5 Software Categories
CategoryTypeExamplesValidation Effort
Cat 1InfrastructureOS, network, antivirusLow — qualification only
Cat 3Non-configured COTSMS Word, Adobe, standard lab instrumentsLow — OQ/vendor assessment
Cat 4Configured COTSLabWare LIMS, Veeva Vault, SAP, TrackwiseMedium-High — full V-model
Cat 5Custom / BespokeIn-house apps, custom scripts for regulated useHighest — full SDLC + validation
Key Point: Most pharma enterprise systems (LIMS, QMS, Veeva) fall under Cat 4. Custom tools built internally are Cat 5 and require the most documentation. A pure off-the-shelf tool used without configuration may be Cat 3.

Step 02 · Onboarding
Validation Planning
& Master Plan
The Validation Master Plan (VMP) or System Validation Plan (SVP) is the governing document. It defines scope, approach, responsibilities, and acceptance criteria before any testing begins.
Validation Plan — Key Components
1
Scope & System Description
System name, version, hosting environment, intended use, regulated functions, interfaces with other systems.
2
Regulatory References
21 CFR Part 11, 21 CFR Part 211, EU GMP Annex 11, ICH Q10, GAMP 5, applicable USP/EP chapters.
3
Validation Approach & Strategy
Risk-based approach statement, GAMP category, IQ/OQ/PQ breakdown, vendor documentation leveraged.
4
Roles & Responsibilities
Validation owner, SMEs, IT, QA reviewer/approver, vendor contacts. RACI matrix often included.
5
Deliverables List
All documents to be produced: URS, FRS, DS, RTM, IQ, OQ, PQ protocols and reports, SOPs, training records.
6
Risk Assessment
System-level risk assessment, GxP risk score, identification of critical functions requiring higher test rigor.
7
Timeline & Milestones
Projected start/end dates, phase gates, QA review checkpoints, go-live date.

Step 03 · Validation Phase
IQ — Installation
Qualification
Verifies that the system is installed correctly and according to vendor specifications, and that the environment meets all prerequisites before functional testing begins.
IQ Protocol Contents
🖥️
Hardware Verification
Server specs vs. vendor minimum requirements. CPU, RAM, disk, network configuration. Rack location, physical access controls.
⚙️
Software Installation
Application version confirmed, installation path documented, license keys recorded, all components installed per runbook.
🔧
Configuration Baseline
Initial configuration screenshots, config files archived, database schema version, service account setup, firewall rules documented.
🌐
Network & Interfaces
Network diagrams, interface connections to other systems, SSL/TLS certificates, port configurations, data flow diagrams.
📁
Backup & DR
Backup schedule configured and tested, disaster recovery procedures documented, RTO/RPO confirmed for the system.
🛡️
Security Configuration
Audit trail enabled and tested, e-signature configuration, password policy settings, inactivity timeout confirmed.
IQ Report — Acceptance Criteria
PASSED: All IQ test steps executed. All actual results match expected. No open critical/major deviations. Deviations (if any) have approved resolutions. IQ Summary Report signed by Validation Owner and QA.
IQ Deliverables
DocumentDescriptionApproval Required
IQ ProtocolPre-approved test plan with all steps and expected resultsValidation Owner + QA
IQ Execution RecordsCompleted test steps with actual results, tester initials, datesTester + Reviewer
Deviation LogAny step that did not pass, root cause, resolutionQA
IQ Summary ReportPass/fail verdict, open items, recommendation to proceed to OQValidation Owner + QA
Step 04 · Validation Phase
OQ — Operational
Qualification
Verifies that the system functions as designed across its full operational range — all configured features, business logic, and system behaviors are tested against the FRS and design specification.
OQ Scope — What Gets Tested
F1
Core Functional Workflows
Every configured business workflow per the FRS. For a LIMS: sample login, analysis assignment, result entry, result review, approval, reporting. For Veeva: document lifecycle (Draft → Review → Approved → Superseded).
F2
User Roles & Permissions
Verify each role can only perform permitted actions. Negative testing critical: an Analyst cannot approve results, a Read-Only user cannot create records.
F3
Electronic Signatures & Audit Trail
E-sig triggers correct (meaning, date/time, user ID). Audit trail captures all creates/edits/deletes. Audit trail is tamper-evident and not modifiable by end users.
F4
Calculations & Business Rules
Configured calculations verified with known inputs/outputs. OOS/OOT flags trigger at correct thresholds. Workflow routing logic verified.
F5
Interface / Integration Testing
Data transfers between integrated systems verified. LIMS ↔ ERP, Veeva ↔ regulatory portals. Verify data completeness, integrity, and error handling.
F6
Negative / Error Path Testing
System behavior on invalid inputs, mandatory field enforcement, duplicate record handling, session timeout, and concurrent user conflicts.
OQ Test Script Structure
Each test script must contain: Test Case ID · Requirement ID (links to RTM) · Objective · Prerequisites / Test Data · Step-by-step actions · Expected Result (pre-defined) · Actual Result (filled during execution) · Pass/Fail · Tester + Date · Reviewer sign-off.
Step 05 · Validation Phase
PQ — Performance
Qualification
Demonstrates that the system consistently performs as intended in the actual operational environment, using real or representative data, by actual end users under real-world conditions.
PQ vs. OQ — Key Difference
⚗️ OQ Focus
Configured features tested by the validation team. Synthetic test data. Controlled environment. Proves the system can do what it's configured to do.
🏭 PQ Focus
End-to-end business processes run by actual users. Production-like or real data. Proves the system will work in day-to-day operation.
PQ Execution Elements
P1
End-to-End Process Scenarios
Complete business scenarios from initiation to final output. E.g., Full sample lifecycle in LIMS from receipt to COA generation. Full document lifecycle in Veeva from draft to regulatory submission.
P2
Volume / Load Testing (if applicable)
If system processes high data volumes, verify performance under expected peak load. Response times documented and compared to acceptance criteria.
P3
Executed by End Users
Business users (not the validation team) run the tests. This also serves as hands-on training. Observed by the validation lead.
P4
Data Integrity Verification
Verify that data entered is accurately stored, retrieved, reported, and archived. ALCOA+ principles confirmed: Attributable, Legible, Contemporaneous, Original, Accurate.
ALCOA+ Principles
A — Attributable
Who performed the action and when. Unique user IDs, no shared logins, audit trail captures username.
L — Legible
Data is readable and permanent. No overwriting without audit trail. Timestamps in unambiguous format.
C — Contemporaneous
Recorded at the time the activity occurs, not retrospectively.
O — Original
First capture is preserved. Source data retained. No transcription errors from paper to electronic.
A — Accurate
Data reflects the actual event. Calculations verified. No rounding errors. Instrument calibration confirmed.
+ Complete, Consistent, Enduring, Available
All data present, time-consistent, retained for required retention period, accessible to authorized users.
Step 06 · Validation Phase
UAT & Requirements
Traceability Matrix
User Acceptance Testing is the final gate before go-live. The RTM ties every requirement to its test evidence. These two together form the core of your validation package.
UAT Process
1
UAT Planning
Business users define acceptance criteria. UAT scripts written based on real workflows. Test environment mirrors production. UAT plan approved by Business Owner and QA.
2
UAT Execution
Business stakeholders execute scenarios. Issues logged in defect tracker with priority (Critical / Major / Minor). No critical or major open defects at go-live.
3
Defect Resolution Cycle
Developer fixes → re-test in same environment → regression testing for impacted areas → QA review of fix. If fix requires configuration change, assess whether re-validation is required.
4
UAT Sign-Off
Business Owner signs UAT completion. All critical/major defects resolved. Minor defects either resolved or accepted with documented justification.
RTM Structure
ColumnContent
URS IDUnique ID from User Requirements Spec (e.g., URS-001)
URS DescriptionThe requirement statement
FRS IDFunctional requirement that fulfills the URS
Design ReferenceWhere in the design spec this is addressed
Test Script ID(s)IQ / OQ / PQ / UAT test case(s) covering this requirement
Test ResultPass / Fail / N/A with report reference
Deviation RefIf failed, cross-reference to deviation record
Audit Rule: 100% of requirements must have test coverage. Any gap requires a documented justification (e.g., "Not Testable — verified via design review"). Auditors specifically look for RTM completeness.

Change Impact Assessment — Key Questions
IA1
What is the nature of the change?
Configuration change? Software upgrade? New interface? SOP update? Infrastructure change? Hardware replacement?
IA2
What GxP functions are affected?
List all impacted validated functions. Assess risk level: High / Medium / Low.
IA3
Does the change require re-validation?
High-risk changes to validated functions typically require OQ/PQ re-execution. Low-risk changes may need only regression testing or impact assessment documentation.
IA4
Does the change require SOP updates?
If business workflows change, affected SOPs must be updated, reviewed, approved, and users retrained before go-live.
IA5
Is rollback possible?
Define rollback plan before implementation. Backup taken and verified. Rollback tested in non-prod if high risk.
Change Categories
CategoryDescriptionRequired Actions
MajorNew GxP functionality, core workflow change, system upgrade, new integrationFull re-validation (OQ/PQ), SOP updates, training, QA approval
ModerateConfiguration update, report change, non-core workflow modificationRegression OQ, impact assessment, QA review
MinorCosmetic UI changes, label updates, non-GxP function changesDocumented impact assessment, approval, no re-validation required
EmergencyCritical system failure, security patch requiring immediate actionExpedited approval path, full retrospective documentation, QA sign-off within defined timeframe
Step 08 · Enhancement Requests
User Enhancement
Request Process
Enhancement requests (ERs) are formal requests for new features or functional improvements from end users. Unlike bug fixes, ERs represent planned changes that enhance system capabilities and must follow a structured lifecycle.
ER Intake & Prioritization
1
Enhancement Request Submission
User submits via formal channel (QMS, ITSM like ServiceNow, or change management module). ER must include: business justification, current state vs. desired state, regulatory impact if any, priority (High/Med/Low), and requesting department.
2
Business Owner Review
Business Owner confirms business need, checks for duplicates or related open changes, assigns initial priority.
3
Technical Feasibility Assessment
System/IT Owner assesses: Is this configurable or does it require custom development? Estimated effort. Dependencies. Impact on existing validated functions.
4
GxP / Validation Impact Assessment
Validation team or QA determines: Does this affect validated functions? What re-testing is required? RTM update needed? SOP/training update required?
5
Change Control Board (CCB) Approval
CCB reviews ER with impact assessment. Approves for implementation with defined scope, timeline, and re-validation requirements. Formally opens a Change Control record.

Environment Strategy: GxP systems must have separate DEV / TEST / UAT / PROD environments. Changes must never be made directly in PROD without following the promotion workflow. Environment qualification (EQ) should be documented for each environment.
Vendor-Managed Systems (e.g., Veeva Vault)
SaaS / Cloud considerations: For vendor-managed systems (Veeva, Salesforce, cloud LIMS), vendor patches and upgrades are pushed on a schedule. You must: (1) monitor vendor release notes, (2) assess GxP impact of each release, (3) run regression testing in sandbox before accepting in production, (4) maintain documentation of your assessment and testing. This is critical — vendor upgrades ARE changes to your validated system.
Step 09 · Ongoing Operations
Periodic Review &
Revalidation
Validated GxP systems require ongoing oversight. Periodic review confirms the system remains in a validated state, identifies configuration drift, and captures the cumulative impact of changes since last review.
Periodic Review Components
PR1
Review Frequency
Typically annual, though risk-based frequency is acceptable. High-criticality systems (e.g., batch release LIMS) reviewed annually. Lower-risk systems may be 18–24 months with justification.
PR2
Change History Review
Review all changes implemented since last periodic review. Confirm all changes went through change control. No undocumented changes (configuration drift check).
PR3
Incident / Deviation Review
Review all system-related incidents, deviations, and CAPAs. Trends identified. Open CAPAs assessed for resolution status.
PR4
User Access Review
Review all active user accounts. Confirm roles are appropriate. Remove or disable stale accounts (terminated employees, role changes). Review privileged access.
PR5
Validation Status Confirmation
Declare the system remains in a validated state (or identify what re-validation is needed). Sign-off by System Owner, Business Owner, and QA.
PR6
SOP Effectiveness Review
Are SOPs being followed? Are there recurring issues that suggest SOP gaps? Confirm SOP revision dates are current.
Revalidation Triggers
⚠️ Always Requires Re-validation
  • Major software version upgrade
  • Migration to new hardware or cloud platform
  • Major new functionality for GxP use
  • Change to audit trail mechanism
  • Change to e-signature implementation
📋 Risk-Based Assessment
  • Minor version / patch upgrades
  • Configuration changes in GxP areas
  • OS or database upgrades
  • New interface to another system
  • Significant SOP changes
Step 10 · Ongoing Operations
SOPs, Training
& Competency
SOPs and training records are not administrative overhead — they are part of the validated state. An auditor will check that users are trained, SOPs are current, and training is documented before looking at anything else.
Required SOPs for a Validated System
👤
Access Control SOP
User provisioning, role assignment, access removal (leavers/movers), periodic access review, emergency access procedure.
🔄
Change Control SOP
How changes are requested, assessed, approved, implemented, and documented. Environment promotion workflow.
💾
Backup & Recovery SOP
Backup schedule, verification, restoration testing frequency, off-site storage, recovery time objective.
🚨
Incident Management SOP
How system incidents are logged, triaged, investigated, and resolved. Link to deviation management system.
📊
Periodic Review SOP
Frequency, scope, responsibilities, and documentation requirements for periodic review of validated systems.
📋
System Use SOPs
Work instructions for each GxP workflow: how to enter data, review, approve, generate reports, handle errors.
Training Program Requirements
T1
Role-Based Training Curriculum
Each role has a defined training curriculum. Analysts train on different SOPs than Reviewers or Admins. Training matrix maintained.
T2
Training Before System Access
Users must complete required training and demonstrate competency before being granted access. Training records must be complete before access request is approved.
T3
Retraining on Change
When SOPs are updated due to system changes, affected users must be retrained. Go-live of a change cannot precede completion of retraining.
T4
Training Record Retention
Training records retained for the life of the system plus applicable regulatory retention period. Typically electronic, stored in LMS or QMS with audit trail.
Step 11 · Ongoing Operations
Audit Readiness &
21 CFR Part 11
Be inspection-ready at all times. FDA, EMA, and internal auditors will examine your validation documentation, change history, user access records, and audit trail integrity.
21 CFR Part 11 / Annex 11 Requirements
RequirementWhat Auditors Check
Audit TrailEnabled for all GxP records. Captures who, what, when, original value, new value. Not editable by users. Reviewed regularly.
Electronic SignaturesUnique to individual, not shared. Linked to signed record. Signatories have signed their Part 11 commitment statement. Includes meaning, date/time.
Access ControlsUnique user IDs. No shared accounts. Appropriate role-based access. Terminated user accounts disabled promptly.
ValidationSystem validated before use. Validation documentation current and complete. Validation status re-confirmed after changes.
System ControlsPassword aging, inactivity timeout, login attempt limits. Session locking. Backup and recovery tested.
Copies of RecordsAbility to generate accurate and complete copies of records in both human-readable and electronic form (for inspection).
Audit-Ready Document Checklist
  • Current Validation Master Plan
  • Complete Validation Package (IQ/OQ/PQ)
  • Current RTM showing 100% coverage
  • All Change Control records (open + closed)
  • Current user access list with roles
  • Periodic Review report (most recent)
  • All current SOPs (approved, in-date)
  • Training records for all current users
  • Incident / deviation log
  • Audit trail review records
  • Backup test records
  • Vendor qualification / audit records
Step 12 · End of Life
System Retirement &
Decommissioning
Decommissioning a GxP system is a formal, validated activity. Data must be migrated or archived in a retrievable format for the required regulatory retention period before the system is shut down.
Decommissioning Plan — Key Elements
D1
Retirement Decision & Justification
Formal decision with business rationale. New system to replace (if applicable) must be fully validated before old system is decommissioned.
D2
Data Retention Assessment
Identify all regulated records in the system. Determine regulatory retention requirements (FDA: batch records 3 yrs post expiry, for example). Define archive format and access method for the retention period.
D3
Data Migration or Archive
Migrate to new system OR archive to validated archive solution. Data migration validation required if migrating to a new GxP system (verify completeness, integrity, no corruption).
D4
Validation Package Archive
All validation documentation (IQ/OQ/PQ, SOPs, change records, training, periodic reviews) must be archived and retrievable for the retention period, even after the system is gone.
D5
Decommissioning Report
Formal report confirming: data migrated/archived successfully, system shut down, licenses returned, infrastructure decommissioned, documentation archived. Signed by System Owner, Business Owner, and QA.
Key Principle: You must be able to reproduce a GxP record from a decommissioned system during a regulatory inspection 10+ years later. Plan your archival strategy accordingly — including the ability to render archived data in human-readable form even when the original software no longer exists.
